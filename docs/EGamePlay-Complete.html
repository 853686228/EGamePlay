Chapter 01

EcsNode — ECS 框架底座

理解 EGamePlay 一切运行机制的基础。EcsNode 是一个面向游戏业务的 OOP-ECS 框架，不追求 DOTS 级性能，而是追求代码可读性和灵活的业务建模能力。

1.1 概述与定位

核心定位
EcsNode 不是 Unity DOTS/ECS。它是一个 OOP 风格的轻量 ECS 框架，核心特点：
• Entity 有类型（继承 EcsEntity），不是纯 ID
• System 通过反射自动注册，无需手动绑定
• 数据和逻辑分离：Model 层只存数据，System 层执行逻辑
• 支持 Entity 父子层级，形成所有权树

核心设计思想
数据与逻辑分离
Entity/Component 只存数据（Model 层），System 包含所有逻辑。同一份数据可以有不同的 System 处理（如 Server 和 Client 用不同的 DamageSystem）。
反射自动注册
System 只需继承 AEntitySystem<T> 并实现对应接口，EcsNode 启动时自动扫描注册。零配置。
层级所有权
Entity 通过 Parent/Children 形成树状结构。父 Entity 销毁时，子树自动清理。CombatEntity → Ability → Effect 就是这样的层级。

1.2 EcsEntity — 实体基类

一切战斗实体（CombatEntity、Ability、AbilityEffect、AbilityTrigger、各种 Action）都继承自 EcsEntity。它提供三个核心能力：身份标识、组件容器、层级关系。

完整类结构

EcsEntity.cs
public class EcsEntity : EcsObject
{
    // ═══════════ 身份标识 ═══════════
    public long Id { get; set; }          // 业务 ID（如技能配置表 ID）
    public long InstanceId { get; set; }  // 运行时唯一实例 ID（由 EcsNode 生成）

    // ═══════════ 组件容器 ═══════════
    public Dictionary<Type, EcsComponent> Components { get; }
        = new Dictionary<Type, EcsComponent>();

    // ═══════════ 层级关系 ═══════════
    public EcsEntity Parent { get; set; }
    public Dictionary<long, EcsEntity> Id2Children { get; }
        = new Dictionary<long, EcsEntity>();

    // ═══════════ 启用状态 ═══════════
    private bool _enable;
    public bool Enable
    {
        get => _enable;
        set {
            if (_enable == value) return;
            _enable = value;
            if (value)
            {
                // 触发 IEnable System
                EcsNode.Instance.DriveEntitySystems(this, typeof(IEnable));
                // 级联启用所有组件
                foreach (var comp in Components.Values)
                    comp.Enable = true;
            }
            else
            {
                // 触发 IDisable System
                EcsNode.Instance.DriveEntitySystems(this, typeof(IDisable));
                foreach (var comp in Components.Values)
                    comp.Enable = false;
            }
        }
    }
}
设计洞察 — Enable 级联
Entity 的 Enable 属性设置时会自动级联到所有挂载的 Component。当 Ability.Enable = false 时，它身上的 EffectDamageComponent、EffectCureComponent 等全部被禁用。这让 Buff 的「暂停/恢复」变成一行代码。

子实体创建与删除
// 创建子实体 — 最常用的 API
public T AddChild<T>() where T : EcsEntity, new()
{
    var entity = new T();
    entity.Id = EcsNode.Instance.NewInstanceId();   // 自动分配唯一 ID
    entity.InstanceId = entity.Id;
    entity.Parent = this;                          // 设置父子关系
    Id2Children.Add(entity.Id, entity);
    EcsNode.Instance.DriveEntitySystems(entity, typeof(IAwake)); // 触发 IAwake
    return entity;
}

// 带初始化回调的版本 — 可在 Awake 之前设置属性
public T AddChild<T>(Action<T> initAction) where T : EcsEntity, new()
{
    var entity = new T();
    entity.Id = EcsNode.Instance.NewInstanceId();
    entity.InstanceId = entity.Id;
    entity.Parent = this;
    Id2Children.Add(entity.Id, entity);
    initAction?.Invoke(entity);                    // ← 在 Awake 前执行初始化
    EcsNode.Instance.DriveEntitySystems(entity, typeof(IAwake));
    return entity;
}

var ability = entity.AddChild<Ability>(x => x.ConfigObject = configObject);
// ↑ ConfigObject 在 Awake 之前就设好了，所以 AbilitySystem.Awake 里能直接用

public void RemoveChild(EcsEntity child)
{
    Id2Children.Remove(child.Id);
    EcsNode.Instance.DriveEntitySystems(child, typeof(IDestroy)); // 触发 IDestroy
    EcsObject.Destroy(child);  // 递归销毁子树
}

1.3 EcsComponent — 组件基类

组件是纯数据容器。它绑定到一个 Entity，有自己的 Enable/Disable 状态（受宿主 Entity 级联控制）。

EcsComponent.cs
public interface IEcsComponent { }

public class EcsComponent : EcsObject, IEcsComponent
{
    public EcsEntity Entity { get; set; }   // 宿主 Entity

    private bool _enable;
    public bool Enable
    {
        get => _enable;
        set {
            if (_enable == value) return;
            _enable = value;
            if (value)
                EcsNode.Instance.DriveComponentSystems(Entity, this, typeof(IEnable));
            else
                EcsNode.Instance.DriveComponentSystems(Entity, this, typeof(IDisable));
        }
    }
}

// 泛型版本 — 提供类型安全的 Entity 访问
public class EcsComponent<T> : EcsComponent where T : EcsEntity
{
    public new T Entity
    {
        get => base.Entity as T;
        set => base.Entity = value;
    }
}
实际组件示例
EGamePlay 中的组件都很简单，只有属性声明：

EffectDamageComponent → 持有 DamageEffect 配置引用 + DamageValueFormula 字符串
EffectAddBuffComponent → 持有 AddStatusEffect 配置 + Duration 浮点数
AbilityDurationComponent → 持有 GameTimer + Duration

组件永远不包含逻辑方法，所有行为都在对应的 System 中。

// Entity System 示例 — AbilitySystem 处理所有 Ability 类型的 Entity
public class AbilitySystem : AEntitySystem<Ability>,
    IAwake<Ability>,       // Ability 创建时调用
    IEnable<Ability>,      // Ability 启用时调用
    IDisable<Ability>,     // Ability 禁用时调用
    IUpdate<Ability>       // 每帧对每个 Ability 调用
{
    public void Awake(Ability entity) { /* 初始化 */ }
    public void Enable(Ability entity) { /* 启用 */ }
    public void Disable(Ability entity) { /* 禁用 */ }
    public void Update(Ability entity) { /* 每帧 */ }
}

// Component System 示例 — 只在 AbilityEffect 挂了 EffectDamageComponent 时生效
public class EffectDamageSystem : AComponentSystem<AbilityEffect, EffectDamageComponent>,
    IAwake<AbilityEffect, EffectDamageComponent>
{
    public void Awake(AbilityEffect entity, EffectDamageComponent component)
    {
        // 仅当 AbilityEffect 挂了 EffectDamageComponent 时触发
        component.DamageEffect = entity.EffectConfig as DamageEffect;
        component.DamageValueFormula = component.DamageEffect.DamageValueFormula;
    }
}
Entity System vs Component System 选择指南
用 Entity System 当：逻辑与 Entity 的整体生命周期相关（创建/销毁/启用/每帧更新）
用 Component System 当：逻辑只在特定 Component 被挂载时才需要执行（如伤害计算只在 EffectDamageComponent 存在时才有意义）

1.5 DriveType — 生命周期接口

DriveType 是一系列接口，System 通过实现这些接口来声明「我想在什么时机被调用」。EcsNode 在对应时机自动找到并调用匹配的 System。

完整生命周期接口一览

接口触发时机典型用途参数
IAwake<T>Entity/Component 创建时初始化配置加载(T entity)
IInit<T>手动调用 Init() 时延迟初始化（依赖已就绪后）(T entity)
IAfterInit<T>Init 之后初始化收尾(T entity)
IEnable<T>Enable = true 时激活效果/订阅事件(T entity) 或 (T entity, C comp)
IDisable<T>Enable = false 时暂停效果/取消订阅(T entity) 或 (T entity, C comp)
IUpdate<T>每帧（Update）计时器更新/状态检测(T entity)
IFixedUpdate<T>每物理帧物理相关更新(T entity)
ISecondUpdate<T>每秒低频更新（如 DOT tick）(T entity)
IOnChange<T,C>Component 数据变更时响应数据变化(T entity, C comp)
IOnAddComponent<T>Component 被添加时观察组件变化(T entity, EcsComponent comp)
IOnRemoveComponent<T>Component 被移除时清理引用(T entity, EcsComponent comp)
IDestroy<T>Entity/Component 销毁时资源释放/引用清理(T entity) 或 (T entity, C comp)
IPreDestroy<T>Destroy 之前销毁前的最终操作(T entity)

生命周期执行顺序

↓

↓
entity.Init()

↓
IInit → IAfterInit

↓

IAwake vs IInit 的区别
IAwake：在 AddChild() 时自动调用。此时 Entity 刚创建，可能还没有设置所有必要属性。
IInit：在手动调用 entity.Init() 时触发。用于需要确保所有前置条件都满足后再执行的初始化逻辑。

在 EGamePlay 中，AbilitySystem.Attach() 的顺序是：AddChild()（触发 Awake）→ AddComponent() → ability.Init()（触发 Init）。

1.6 EcsNode — 核心调度器

EcsNode 是整个框架的「大脑」。它在启动时通过反射发现所有 System，建立映射表，然后在运行时将 Entity/Component 的生命周期事件分发给匹配的 System。

public void DriveEntitySystems(EcsEntity entity, Type driveType)

        {

            cachedMethod.Invoke(system, new object[] { entity });

// Update 驱动 — 每帧遍历所有需要 Update 的 Entity
public void DriveEntityUpdate()
{
    foreach (var pair in Type2Entities)
    {
        var entityType = pair.Key;
        var entities = pair.Value;
        var systems = EntityType2Systems[entityType];

        foreach (var entity in entities)
        {
            foreach (var system in systems.OfType<IUpdate>())
            {
                system.Update(entity);
            }
        }
    }
}

1.8 EntityState — 安全引用

当你持有一个 Entity 引用，但那个 Entity 可能随时被销毁（如目标死亡），如何避免空引用？EntityState 解决这个问题。

EntityState.cs
public struct EntityState<T> where T : EcsEntity
{
    private T _entity;
    private long _instanceId;  // 创建时记录的 InstanceId

    public EntityState(T entity)
    {
        _entity = entity;
        _instanceId = entity?.InstanceId ?? 0;
    }

    // 隐式转换 — 自动检测实体是否还有效
    public static implicit operator T(EntityState<T> state)
    {
        if (state._entity == null) return null;
        if (state._entity.IsDisposed) return null;
        // InstanceId 不匹配说明原 Entity 已销毁，新 Entity 复用了内存
        if (state._entity.InstanceId != state._instanceId) return null;
        return state._entity;
    }
}

// 使用示例：
EntityState<CombatEntity> targetRef = new(target);
// ... 若干帧后 ...
CombatEntity t = targetRef;  // 如果 target 已死，t == null
if (t != null) { /* 安全操作 */ }
为什么不用 WeakReference
C# 的 WeakReference 基于 GC 判断对象存活，但 EcsEntity 的「逻辑死亡」（被战斗系统销毁）和 GC 回收时机不同步。EntityState 通过 InstanceId 比对实现了逻辑层面的安全引用，不依赖 GC。

1.9 设计总结

为什么不用纯 ECS (DOTS)?
DOTS 的 Archetype 内存布局对技能系统帮助有限 — 技能逻辑复杂但实体数量少（一场战斗几十个 Entity）。OOP-ECS 换来了更好的调试体验和更自然的业务建模。
反射的性能影响
反射只在启动时执行一次（注册 System、缓存 MethodInfo）。运行时的 DriveEntitySystems 使用缓存的方法引用，实际开销接近直接调用。
所有权 = 生命周期
Parent-Children 关系不只是组织结构，更是生命周期管理。CombatEntity 死亡 → 所有 Ability 销毁 → 所有 Effect/Trigger 销毁。零手动清理。
已知局限
• 大量短生命周期 Entity 的 GC 压力（Action 每次触发都 new）
• 无对象池
• System 无法动态增删（启动时注册，运行时固定）

Chapter 02

FloatNumeric — 数值系统

战斗数值的一切基础。HP、攻击力、防御、暴击率、移速...所有可变属性都通过 FloatNumeric 管理。它的核心设计目标：所有修改可逆 — Buff 结束时移除 Modifier，数值精确恢复。

2.1 概述

核心问题
一个角色的攻击力 = 基础值 + 装备加成 + Buff 加成 + 被动技能 + ...。当某个 Buff 结束时，如何精确回退它的效果？

错误做法： atk = atk - buffValue — 浮点误差累积，且百分比加成无法简单减回来
正确做法： 每个修改源持有自己的 Modifier 引用，移除时从集合中删除，重新计算最终值

2.2 FloatModifier — 修改器

FloatModifier 是最小单元 — 一个可变的浮点数值。每次 Buff 给角色加攻击力时，就创建一个 FloatModifier 并添加到对应集合中。

FloatModifier.cs
namespace EGamePlay.Combat
{
    /// 单个修改器 — 代表一个修改源的贡献值
    public class FloatModifier
    {
        public float Value;  // 修改值（如 +20 攻击力，或 +50% 百分比）
    }

    /// 修改器类型枚举
    public enum ModifierType
    {
        Add,          // 基础加法（第一层）
        PctAdd,       // 百分比加法（第二层）
        FinalAdd,     // 最终加法（第三层）
        FinalPctAdd,  // 最终百分比（第四层）
    }
}
为什么 Modifier 是 class 不是 struct
因为需要引用语义。当 Buff 结束时，我们需要找到并移除「那个特定的 Modifier 实例」，而不是通过值匹配。两个 Buff 可能都加 20 攻击力，但移除其中一个不能影响另一个。

2.3 FloatModifierCollection — 修改器集合

多个 Modifier 组成一个 Collection（如所有加法修改器放一个集合）。集合负责自动维护 TotalValue（所有 Modifier.Value 之和）。

FloatModifier.cs（续）
public class FloatModifierCollection
{
    /// 当 TotalValue 变化时通知外层 FloatNumeric 重算
    public event Action UpdateEvent;

    /// 所有修改器的值之和
    public float TotalValue { get; private set; }

    /// 内部存储
    private List<FloatModifier> _modifiers = new();

    /// 添加修改器
    public void AddModifier(FloatModifier modifier)
    {
        _modifiers.Add(modifier);
        TotalValue += modifier.Value;  // 增量更新，O(1)
        UpdateEvent?.Invoke();          // 通知 FloatNumeric 重算最终值
    }

    /// 移除修改器
    public void RemoveModifier(FloatModifier modifier)
    {
        _modifiers.Remove(modifier);
        TotalValue -= modifier.Value;  // 增量更新，O(1)
        UpdateEvent?.Invoke();
    }

    /// 全量重算（用于修改器值被外部改变后）
    public void Update()
    {
        TotalValue = 0;
        foreach (var m in _modifiers)
            TotalValue += m.Value;
        UpdateEvent?.Invoke();
    }
}
增量更新 vs 全量重算
Add/Remove 时直接 ±= 单个值，O(1) 时间复杂度。只有在 Modifier.Value 被外部修改（罕见情况）时才需要全量 Update()。这对于频繁 Buff 增删的 MOBA 场景非常关键。

2.4 FloatNumeric — 完整数值类

FloatNumeric 持有一个 baseValue 和四个 ModifierCollection，任何一个变化都会触发重算最终 Value。

FloatNumeric.cs — 完整源码
namespace EGamePlay.Combat
{
    public class FloatNumeric
    {
        // ═══════════ 最终计算结果 ═══════════
        public float Value { get; private set; }

        // ═══════════ 基础值 ═══════════
        public float baseValue;

        // ═══════════ 四层修改器集合 ═══════════
        public FloatModifierCollection add          = new();  // 加法层
        public FloatModifierCollection pctAdd       = new();  // 百分比层
        public FloatModifierCollection finalAdd     = new();  // 最终加法层
        public FloatModifierCollection finalPctAdd  = new();  // 最终百分比层

        // ═══════════ 类型映射 ═══════════
        public Dictionary<ModifierType, FloatModifierCollection> TypeModifierCollections;

        // ═══════════ 属性类型标识 ═══════════
        public AttributeType AttributeType { get; set; }

        // ═══════════ 构造函数 ═══════════
        public FloatNumeric()
        {
            TypeModifierCollections = new Dictionary<ModifierType, FloatModifierCollection>
            {
                { ModifierType.Add, add },
                { ModifierType.PctAdd, pctAdd },
                { ModifierType.FinalAdd, finalAdd },
                { ModifierType.FinalPctAdd, finalPctAdd },
            };

            // 订阅所有集合的变化事件 → 触发重算
            add.UpdateEvent += Update;
            pctAdd.UpdateEvent += Update;
            finalAdd.UpdateEvent += Update;
            finalPctAdd.UpdateEvent += Update;
        }

        // ═══════════ 核心计算 ═══════════
        public void Update()
        {
            var value1 = baseValue;
            var value2 = (value1 + add.TotalValue) * (100 + pctAdd.TotalValue) / 100;
            var value3 = (value2 + finalAdd.TotalValue) * (100 + finalPctAdd.TotalValue) / 100;
            Value = value3;
        }

        // ═══════════ 设置基础值 ═══════════
        public void SetBase(float value)
        {
            baseValue = value;
            Update();
        }
    }
}

2.5 四层计算公式详解

公式拆解

计算步骤
// 第一层：基础值
value1 = baseValue

// 第二层：基础 + 加法修改，然后乘以百分比
value2 = (value1 + add.TotalValue) × (100 + pctAdd.TotalValue) / 100

// 第三层：上一层结果 + 最终加法，再乘以最终百分比
value3 = (value2 + finalAdd.TotalValue) × (100 + finalPctAdd.TotalValue) / 100

// 最终值
Value = value3

具体计算示例

场景baseValueAddPctAddFinalAddFinalPctAdd结果
纯基础1000000 100
+装备 20 攻10020000 (100+20)×1 = 120
+Buff 50% 攻100205000 (100+20)×1.5 = 180
+光环 +10 最终1002050100 (180+10)×1 = 190
+被动 20% 最终10020501020 (180+10)×1.2 = 228
四层的设计意义
为什么不是单层 List？

Add + PctAdd（基础层）：代表装备/基础 Buff 的加成。百分比是基于「基础+Add」计算的。
→ 意义：一个 +50% Buff 只加成「基础攻击」，不会加成后续的最终加法

FinalAdd + FinalPctAdd（最终层）：代表不受基础百分比影响的加成。
→ 意义：「无视防御的固定穿透伤害」或「最终伤害减免 30%」

实际 MOBA 映射：
• 装备面板攻击力 → Add
• 「攻击力提升 50%」Buff → PctAdd
• 红 Buff 附加 50 真实伤害 → FinalAdd
• 「造成伤害提高 20%」（如输出方的终伤） → FinalPctAdd

可逆性证明

可逆性示例
// 场景：角色有 base=100，装备+20(Add)，Buff+50%(PctAdd)
// 当前值 = (100+20) * 1.5 = 180

// Buff 结束，移除 PctAdd Modifier：
pctAdd.RemoveModifier(buffModifier);  // pctAdd.TotalValue: 50 → 0
// 自动触发 Update():
// value2 = (100+20) * (100+0)/100 = 120 ← 精确回退到 Buff 前

// 对比直接减法：
// 180 - (100+20)*0.5 = 180 - 60 = 120 ← 数学上正确但需要记住原始计算
// 180 - 50% = 90 ← 错误！这是减了 180 的 50%
// 用 Modifier 集合就不会犯这种错误 — Remove 后自动重算

// Buff 生效时（Enable）— 添加 Modifier
public void Enable(AbilityEffect entity, EffectAttributeModifyComponent component)
{
    var parentEntity = entity.OwnerAbility.Parent;  // 获取挂载 Buff 的角色
    var effect = component.AttributeModifyEffect;

    // 解析配置中的数值（如 "50" 或 "30%"）
    var numericValue = component.ModifyValueFormula.Replace("%", "");
    var value = (float)ExpressionHelper.Evaluate(numericValue);

    // 创建修改器实例
    var modifier = new FloatModifier() { Value = value };

    // 找到目标属性（如 Attack）
    var numeric = AttributeSystem.GetNumeric(parentEntity as CombatEntity, effect.AttributeType);

    // 根据修改类型添加到对应层
    if (effect.ModifyType == ModifyType.Add)
        NumericSystem.AddModifier(numeric, ModifierType.FinalAdd, modifier);
    if (effect.ModifyType == ModifyType.PercentAdd)
        NumericSystem.AddModifier(numeric, ModifierType.FinalPctAdd, modifier);

    // ★ 保存 modifier 引用，用于 Disable 时移除
    component.AttributeModifier = modifier;
}

// Buff 失效时（Disable）— 移除同一个 Modifier 实例
public void Disable(AbilityEffect entity, EffectAttributeModifyComponent component)
{
    var numeric = AttributeSystem.GetNumeric(...);
    if (component.AttributeModifyEffect.ModifyType == ModifyType.Add)
        NumericSystem.RemoveModifier(numeric, ModifierType.FinalAdd, component.AttributeModifier);
    if (component.AttributeModifyEffect.ModifyType == ModifyType.PercentAdd)
        NumericSystem.RemoveModifier(numeric, ModifierType.FinalPctAdd, component.AttributeModifier);
    // ↑ 移除的是同一个实例！自动触发重算，数值精确恢复
}

// 配置中可以写表达式，引用属性值作为参数

private static int ParseDamage(AbilityEffect entity)
{

    var expression = ExpressionHelper.ExpressionParser
        .EvaluateExpression(component.DamageValueFormula);

2.8 设计总结

核心不变量
FloatNumeric.Value 永远等于公式计算结果。任何 Modifier 增删都立即触发重算。不存在「脏标记延迟计算」— 简单粗暴但正确。
内存开销
每个 FloatNumeric = 1 float + 4 个 List + 1 个 Dictionary。一个角色 8 个属性 ≈ 8 个 FloatNumeric 实例。对比 MOBA 中几十个角色，总计百级别实例，内存可忽略。
潜在优化点
如果大量 Modifier 同时增删（如群体 Buff），每个 Add/Remove 都触发一次 Update。可考虑批量操作后统一 Update 一次。当前实现足够用。

可拦截：TryMakeAction 可以返回 false（如沉默状态下不能施法）。
可替换：不同的 CombatEntity 可以有不同的 DamageAbility 实现（如 Boss 有特殊伤害逻辑）。

Ability — 技能/Buff 核心系统

EGamePlay 最核心的设计决策之一：技能和 Buff 是同一个东西。Ability 是统一的「能力」抽象，IsBuff 只是一个标志位。这消除了大量重复代码和特殊处理逻辑。

核心定位

4.3 AbilitySystem.Awake — 初始化流程

Awake 是 Ability 创建后立刻执行的初始化逻辑。它从配置表加载数据、解析目标规则、创建子实体（Effect/Trigger），最后加载执行时间轴资源。

AbilitySystem.cs — Awake
public void Awake(Ability entity)
{
    // ① 从配置表加载 AbilityConfig
    entity.Config = AppStatic.Config.Get<AbilityConfig>(entity.ConfigObject.Id);
    entity.Name = entity.Config.Name;

    // ② 解析目标阵营类型
    CheckAffectTargetType(entity);

    // ③ 解析目标选取方式
    CheckTargetSelectType(entity);

    // ④ 创建所有 AbilityEffect 子实体
    foreach (var item in entity.ConfigObject.Effects)
    {
        var abilityEffect = AbilityEffectSystem.Create(entity, item);
        abilityEffect.Init();
        entity.AbilityEffects.Add(abilityEffect);
    }

    // ⑤ 创建所有 AbilityTrigger 子实体
    foreach (var item in entity.ConfigObject.TriggerActions)
    {
        var abilityTrigger = AbilityTriggerSystem.Create(entity, item);
        abilityTrigger.Init();
        entity.AbilityTriggers.Add(abilityTrigger);
    }

    // ⑥ 加载执行时间轴
    LoadExecution(entity);
}

AbilityDurationSystem — 持续时间管理

AbilityDurationSystem.cs — 完整源码
public class AbilityDurationSystem : AComponentSystem<Ability, AbilityDurationComponent>,
    IAwake<Ability, AbilityDurationComponent>
{
    public void Awake(Ability entity, AbilityDurationComponent component)
    {
        var lifeTime = component.Duration;
        component.LifeTimer = new GameTimer(lifeTime);
    }

    public static void Update(Ability entity, AbilityDurationComponent component)
    {
        if (component.LifeTimer.IsRunning)
        {
            component.LifeTimer.UpdateAsFinish(Time.deltaTime, () => OnTimeEnd(entity));
        }
    }

    public static void OnTimeEnd(Ability ability)
    {
        var config = ability.Config;
        if (config.Type == "Skill")
        {
            SkillSystem.Remove(ability.ParentEntity as CombatEntity, ability);
        }
        if (config.Type == "Buff")
        {
            BuffSystem.Remove(ability.ParentEntity as CombatEntity, ability);
        }
    }
}
持续时间是可选的
AbilityDurationComponent 只在需要时挂载。被动技能永远不挂它（永久生效），Buff 在 Attach 时根据配置挂载。
AbilitySystem.Update 用 is { } component 模式匹配安全获取 — 如果没有这个组件，整个 Update 就是空操作。
零特殊情况，纯组合式设计。

4.5 Attach / Remove — 挂载与移除

Ability 通过 AbilitySystem.Attach 挂载到 CombatEntity 上，通过 AbilitySystem.Remove 移除。这是外部代码与 Ability 系统交互的唯一入口。

AbilitySystem.cs — Attach
/// 挂载能力，技能、被动、buff等都通过这个接口挂载
public static Ability Attach(CombatEntity entity, object configObject)
{
    // ① 获取 AbilityComponent（存放所有 Ability 的容器）
    var component = entity.GetComponent<AbilityComponent>();

    // ② 创建 Ability 子实体，在 Awake 前注入 ConfigObject
    var ability = entity.AddChild<Ability>(x => x.ConfigObject = configObject as AbilityConfigObject);
    // ↑ AddChild 触发 AbilitySystem.Awake → 加载配置、创建子实体

    // ③ 添加等级组件
    ability.AddComponent<AbilityLevelComponent>();

    // ④ 手动触发 Init（延迟初始化）
    ability.Init();

    // ⑤ 注册到 AbilityComponent 的 ID 映射表
    component.IdAbilities.Add(ability.Id, ability);

    return ability;
}

AbilitySystem.cs — Remove
public static void Remove(CombatEntity entity, Ability ability)
{
    // ① 从注册表移除
    var component = entity.GetComponent<AbilityComponent>();
    component.IdAbilities.Remove(ability.Id);

    // ② 结束能力（Disable + Destroy）
    EndAbility(ability);
}

public static void EndAbility(Ability entity)
{
    DeactivateAbility(entity);   // Enable = false → 触发所有 IDisable
    EcsObject.Destroy(entity);    // 销毁 Entity + 递归销毁子树
}
Attach 执行序列详解
Attach(entity, config) 一调用，以下事件按顺序发生：

1. AddChild<Ability>() → 创建 Ability 实例，设置 Parent
2. initAction 执行 → ConfigObject = configObject
3. EcsNode 触发 AbilitySystem.Awake() → 加载 Config、创建 Effects/Triggers
4. AddComponent<AbilityLevelComponent>() → 挂载等级组件
5. ability.Init() → 触发 IInit System（如有）
6. 注册到 IdAbilities 字典

注意 Awake 在第 3 步，此时 ConfigObject 已在第 2 步设好。initAction 的存在确保了 Awake 能安全读取配置。

4.6 Skill vs Buff — 统一设计

这是整个框架最重要的架构决策：技能和 Buff 使用完全相同的 Ability 类。差异仅通过配置和组件组合来区分。

技能 (Skill)
• Config.Type == "Skill"
• 有 ExecutionObject（技能表现时间轴）
• 有目标选取（CollisionSelect / PlayerSelect）
• 通常通过 Trigger 触发 Effect
• 可能有冷却（CooldownTimer）
• 可能有持续时间（引导型技能）
Buff / Debuff
• Config.Type == "Buff"
• 通常无 ExecutionObject
• 目标 = 自身（TargetGroup = "自身"）
• Enable 时 Effect 直接生效（属性修改/行为控制）
• 几乎总有 AbilityDurationComponent
• 持续时间结束自动 Remove 
统一机制 — IsBuff 只是一个标志
// Ability.cs 中只有这一行区分 Skill 和 Buff：
public bool IsBuff => Config.Type == "Buff";

// AbilityDurationSystem.OnTimeEnd — 结束时根据类型调用不同的 Remove
if (config.Type == "Skill")
    SkillSystem.Remove(ability.ParentEntity as CombatEntity, ability);
if (config.Type == "Buff")
    BuffSystem.Remove(ability.ParentEntity as CombatEntity, ability);

// SkillSystem.Remove 和 BuffSystem.Remove 最终都调用 AbilitySystem.Remove
// 区别仅在于它们各自的事件通知（如 UI 刷新、Buff 图标移除等）
为什么这个设计是对的
传统做法是 Skill 和 Buff 各有一套类体系，导致：
• 伤害效果写两套（技能伤害 + Buff 伤害）
• 属性修改写两套
• 持续时间管理写两套
• 触发条件写两套

统一为 Ability 后，所有这些都只需要一套实现。Buff 给目标加状态？那就是一个 EffectAddBuff，和技能施加 Buff 走完全相同的代码路径。

4.8 设计总结

统一抽象的代价
Skill 和 Buff 共享一个类意味着 Ability 上可能存在对当前类型无意义的属性（如 Buff 的 CooldownTimer 永远不用）。这点内存浪费换来的是代码复用和架构简洁，是值得的。
组合式能力声明
一个 Ability 的行为完全由它的 Effects 和 Triggers 列表决定。同样的 EffectDamageComponent 可以出现在技能的 Effect 上，也可以出现在 Buff 的 Effect 上。零代码重复。
Enable 是核心机制
整个系统的激活/停用都通过 Enable 属性驱动。一行 ability.Enable = false 触发完整的回退链：Disable System → 移除 Modifier → 数值恢复。极简的 API，强大的效果。
已知局限
• 字符串类型判断（"Skill"/"Buff"）缺少编译期安全
• 没有 Ability 堆叠机制（同名 Buff 叠加需要额外处理）

Chapter 05

AbilityEffect — 效果系统

Effect 是 Ability 的「做什么」。一个技能/Buff 的实际效果——伤害、治疗、加 Buff、修改属性、控制行为、护盾——全部通过 AbilityEffect 子实体 + 对应 Component + System 来实现。

5.1 概述

核心架构
AbilityEffect 是一个 Entity，挂载在 Ability 下。它通过 组件组合 声明自己的类型：
• 挂了 EffectDamageComponent → 这是一个伤害效果
• 挂了 EffectCureComponent → 这是一个治疗效果
• 挂了 EffectAddBuffComponent → 这是一个施加 Buff 效果

每种 Component 有对应的 System 处理具体逻辑。Effect 本身不包含任何业务代码。

关键模式：Effect 是配置声明，Action 是运行时执行。 Effect 描述「能造成什么」，Action 是「真正造成了什么」。

AbilityEffect.cs — 完整源码
namespace EGamePlay.Combat
{
    public struct TriggerContext
    {
        public AbilityTrigger AbilityTrigger;   // 触发此效果的 Trigger
        public EcsEntity TriggerSource;        // 触发源
        public AbilityItem AbilityItem;        // 技能投射物（如子弹）

设计洞察 — 向上导航
OwnerAbility 通过 GetParent<Ability>() 获得（层级关系）
OwnerEntity 通过 OwnerAbility.OwnerEntity 再向上跳一层

这意味着从任何一个 AbilityEffect 实例，你可以沿着父子链向上找到持有它的 Ability 和 CombatEntity。无需额外存储引用。

5.3 AbilityEffectSystem.Create — 工厂方法

Create 是 AbilityEffect 的工厂方法。它根据配置类型动态挂载对应的 Component，实现了配置驱动的组件组合。

AbilityEffectSystem.cs — Create 工厂
public static AbilityEffect Create(Ability ability, Effect effectConfig)
{
    var abilityEffect = ability.AddChild<AbilityEffect>();
    abilityEffect.EffectConfig = effectConfig;
    abilityEffect.Name = effectConfig.GetType().Name;

    /// 行动控制
    if (effectConfig is ActionControlEffect)
        abilityEffect.AddComponent<EffectActionControlComponent>();

    /// 属性修改
    if (effectConfig is AttributeModifyEffect)
        abilityEffect.AddComponent<EffectAttributeModifyComponent>();

    /// 伤害效果
    if (effectConfig is DamageEffect)
        abilityEffect.AddComponent<EffectDamageComponent>();

    /// 治疗效果
    if (effectConfig is CureEffect)
        abilityEffect.AddComponent<EffectCureComponent>();

    /// 护盾防御效果
    if (effectConfig is ShieldDefenseEffect)
        abilityEffect.AddComponent<EffectShieldDefenseComponent>();

    /// 施加状态效果
    if (effectConfig is AddStatusEffect)
        abilityEffect.AddComponent<EffectAddBuffComponent>();

    /// 效果装饰器
    var decorators = abilityEffect.EffectConfig.Decorators;
    if (decorators != null && decorators.Count > 0)
        abilityEffect.AddComponent<EffectDecoratorComponent>();

    return abilityEffect;
}
类型判断用 is 而非字符串
注意这里用的是 C# 的 is 类型检测（effectConfig is DamageEffect），而不是字符串比较。
配置类型是继承自 Effect 基类的具体子类，编译期类型安全。

这与 AbilitySystem 中用字符串 "Buff"/"Skill" 判断不同 — 因为 Effect 类型是由程序员定义的代码类型，而 Skill/Buff 的分类是策划配表决定的。
用 if 不用 switch/else if 的原因
注意这里不是 else if 而是独立的 if。这意味着一个 Effect 理论上可以同时挂多个 Component（如果配置类型有多重继承关系）。虽然当前配置类型不会同时满足多个条件，但这种写法保留了扩展性。

5.4 TriggerApply — 效果分发

当 Trigger 触发时，调用 AbilityEffectSystem.TriggerApply。它遍历 AbilityEffect 的所有 Component，找到对应的 System 调用其 TriggerApply。

AbilityEffectSystem.cs — TriggerApply 分发
public static void TriggerApply(EffectAssignAction effectAssign)
{
    var abilityEffect = effectAssign.AbilityEffect;

    // 遍历 AbilityEffect 身上的所有组件
    foreach (var item in effectAssign.AbilityEffect.Components.Values)
    {
        if (item is EffectActionControlComponent)
            EffectActionControlSystem.TriggerApply(abilityEffect, effectAssign);

        if (item is EffectAddBuffComponent)
            EffectAddBuffSystem.TriggerApply(abilityEffect, effectAssign);

        if (item is EffectAttributeModifyComponent)
            EffectAttributeSystem.TriggerApply(abilityEffect, effectAssign);

        if (item is EffectCureComponent)
            EffectCureSystem.TriggerApply(abilityEffect, effectAssign);

        if (item is EffectDamageComponent)
            EffectDamageSystem.TriggerApply(abilityEffect, effectAssign);

        if (item is EffectShieldDefenseComponent)
            EffectShieldSystem.TriggerApply(abilityEffect, effectAssign);
    }
}
EffectAssignAction 是什么
EffectAssignAction 是一个 Action 实体（Ch08 详述），携带了触发上下文：
• .AbilityEffect — 哪个 Effect 被触发
• .Target — 效果作用目标
• .SourceAbility — 来源 Ability

它是 Effect（配置声明）和 Action（实际执行）之间的桥梁。

5.5 EffectDamageSystem — 伤害效果

伤害是最常用的效果类型。它解析公式计算伤害值，然后通过 DamageAbility 创建 DamageAction 执行实际伤害。

EffectDamageSystem.cs — 完整源码
public class EffectDamageSystem : AComponentSystem<AbilityEffect, EffectDamageComponent>,
    IAwake<AbilityEffect, EffectDamageComponent>
{
    public void Awake(AbilityEffect entity, EffectDamageComponent component)
    {
        component.DamageEffect = entity.EffectConfig as DamageEffect;
        component.DamageValueFormula = component.DamageEffect.DamageValueFormula;
    }

    public static int GetDamageValue(AbilityEffect entity)
    {
        return ParseDamage(entity);
    }

    private static int ParseDamage(AbilityEffect entity)
    {
        var component = entity.GetComponent<EffectDamageComponent>();
        var expression = ExpressionHelper.ExpressionParser
            .EvaluateExpression(component.DamageValueFormula);

        // 注入运行时属性值到表达式参数
        if (expression.Parameters.ContainsKey("攻击力数值"))
        {
            expression.Parameters["攻击力数值"].Value =
                entity.OwnerEntity.GetComponent<AttributeComponent>().Attack.Value;
        }

        return (int)System.Math.Ceiling((float)expression.Value);
    }

    public static void TriggerApply(AbilityEffect entity, EffectAssignAction effectAssign)
    {
        var component = entity.GetComponent<EffectDamageComponent>();
        if (entity.OwnerEntity.DamageAbility.TryMakeAction(out var damageAction))
        {
            damageAction.SourceAssignAction = effectAssign;
            damageAction.Target = effectAssign.Target;
            damageAction.DamageSource = DamageSource.Skill;
            DamageActionSystem.Execute(damageAction);
        }
    }
}
公式驱动的伤害计算
DamageValueFormula 是配置中的字符串表达式（如 "攻击力数值 * 2.5 + 100"）。
ParseDamage 在运行时解析表达式、注入当前属性值、计算结果。

这意味着策划可以在 Excel 中写任意伤害公式，无需程序修改代码。
属性值来自 FloatNumeric.Value（Ch02），已经包含所有 Buff 加成。

5.6 EffectCureSystem — 治疗效果

EffectCureSystem.cs — 完整源码
public class EffectCureSystem : AComponentSystem<AbilityEffect, EffectCureComponent>,
    IAwake<AbilityEffect, EffectCureComponent>
{
    public void Awake(AbilityEffect entity, EffectCureComponent component)
    {
        component.CureEffect = entity.EffectConfig as CureEffect;
        component.CureValueProperty = component.CureEffect.CureValueFormula;
    }

    public static int GetCureValue(AbilityEffect entity)
    {
        return ParseValue(entity);
    }

    private static int ParseValue(AbilityEffect entity)
    {
        var expression = ExpressionHelper.ExpressionParser
            .EvaluateExpression(entity.GetComponent<EffectCureComponent>().CureValueProperty);

        if (expression.Parameters.ContainsKey("生命值数值"))
        {
            expression.Parameters["生命值数值"].Value =
                entity.OwnerEntity.GetComponent<AttributeComponent>().HealthPoint.Value;
        }

        return (int)System.Math.Ceiling((float)expression.Value);

    public static void TriggerApply(AbilityEffect entity, EffectAssignAction effectAssign)
    {
        if (entity.OwnerEntity.CureAbility.TryMakeAction(out var action))
        {
            action.SourceAssignAction = effectAssign;
            action.Target = effectAssign.Target;
            CureActionSystem.Execute(action);
        }
    }
}
治疗与伤害的对称结构
EffectCureSystem 和 EffectDamageSystem 结构完全对称：
• Awake：缓存配置引用 + 公式字符串
• ParseValue：解析表达式 + 注入属性值
• TriggerApply：创建 Action + Execute

区别仅在于使用的 Ability（CureAbility vs DamageAbility）和 ActionSystem（CureActionSystem vs DamageActionSystem）。

5.8 EffectAttributeSystem — 属性修改

属性修改是 Buff 最常见的效果。它在 Enable 时添加 Modifier，Disable 时移除 Modifier，利用 FloatNumeric 的可逆性实现精确回退。

EffectAttributeSystem.cs — 完整源码
public class EffectAttributeSystem : AComponentSystem<AbilityEffect, EffectAttributeModifyComponent>,
    IAwake<AbilityEffect, EffectAttributeModifyComponent>,
    IEnable<AbilityEffect, EffectAttributeModifyComponent>,
    IDisable<AbilityEffect, EffectAttributeModifyComponent>
{
    public void Awake(AbilityEffect entity, EffectAttributeModifyComponent component)
    {
        component.AttributeModifyEffect = entity.EffectConfig as AttributeModifyEffect;
        component.ModifyValueFormula = component.AttributeModifyEffect.NumericValue;
    }

    public void Enable(AbilityEffect entity, EffectAttributeModifyComponent component)
    {
        var parentEntity = entity.OwnerAbility.Parent;
        var attributeModifyEffect = component.AttributeModifyEffect;
        var numericValue = component.ModifyValueFormula;
        numericValue = numericValue.Replace("%", "");

        var expression = ExpressionHelper.ExpressionParser.EvaluateExpression(numericValue);
        var value = (float)expression.Value;
        var numericModifier = new FloatModifier() { Value = value };

        var attributeType = attributeModifyEffect.AttributeType;
        var numeric = AttributeSystem.GetNumeric(parentEntity as CombatEntity, attributeType);

        if (attributeModifyEffect.ModifyType == ModifyType.BaseValue)
        {
            NumericSystem.SetBase(numeric, value);
        }
        if (attributeModifyEffect.ModifyType == ModifyType.Add)
        {
            NumericSystem.AddModifier(numeric, ModifierType.FinalAdd, numericModifier);
        }
        if (attributeModifyEffect.ModifyType == ModifyType.PercentAdd)
        {
            NumericSystem.AddModifier(numeric, ModifierType.FinalPctAdd, numericModifier);
        }

        component.AttributeModifier = numericModifier;  // ★ 保存引用用于 Disable 时移除
    }

    public void Disable(AbilityEffect entity, EffectAttributeModifyComponent component)
    {
        var parentEntity = entity.OwnerAbility.Parent;
        var attributeType = component.AttributeModifyEffect.AttributeType;
        var numeric = AttributeSystem.GetNumeric(parentEntity as CombatEntity, attributeType);

        if (component.AttributeModifyEffect.ModifyType == ModifyType.Add)
        {
            NumericSystem.RemoveModifier(numeric, ModifierType.FinalAdd,
                component.AttributeModifier);
        }
        if (component.AttributeModifyEffect.ModifyType == ModifyType.PercentAdd)
        {
            NumericSystem.RemoveModifier(numeric, ModifierType.FinalPctAdd,
                component.AttributeModifier);
        }
    }

    public static void TriggerApply(AbilityEffect entity, EffectAssignAction effectAssign)
    {
        entity.GetComponent<EffectAttributeModifyComponent>().Enable = true;
    }
}
完美闭环 — Enable/Disable 对称性
这是整套 ECS 架构的精髓展示：

Enable 时： 解析配置 → 创建 FloatModifier → AddModifier → 数值生效 → 保存 Modifier 引用
Disable 时： 取出保存的 Modifier 引用 → RemoveModifier → 数值精确恢复

Buff 结束 → Ability.Enable = false → 级联 → AbilityEffect.Enable = false → 级联 → Component.Enable = false → 触发 EffectAttributeSystem.Disable → 移除 Modifier。

整条链自动驱动，零手动管理。

5.9 EffectActionControlSystem — 行为控制

行为控制效果（眩晕、沉默、击飞等）在 Enable/Disable 时通知 BuffSystem 重新计算角色的控制状态。

EffectActionControlSystem.cs — 完整源码
public class EffectActionControlSystem : AComponentSystem<AbilityEffect, EffectActionControlComponent>,
    IAwake<AbilityEffect, EffectActionControlComponent>,
    IEnable<AbilityEffect, EffectActionControlComponent>,
    IDisable<AbilityEffect, EffectActionControlComponent>
{
    public void Awake(AbilityEffect entity, EffectActionControlComponent component)
    {
        component.ActionControlEffect = entity.EffectConfig as ActionControlEffect;
    }

    public void Enable(AbilityEffect entity, EffectActionControlComponent component)
    {
        BuffSystem.OnBuffChanged(entity.Parent.Parent as CombatEntity,
            entity.GetParent<Ability>());
    }

    public void Disable(AbilityEffect entity, EffectActionControlComponent component)
    {
        BuffSystem.OnBuffChanged(entity.Parent.Parent as CombatEntity,
            entity.GetParent<Ability>());
    }

    public static void TriggerApply(AbilityEffect entity, EffectAssignAction effectAssign)
    {
        entity.GetComponent<EffectActionControlComponent>().Enable = true;
    }
}
BuffSystem.OnBuffChanged 的作用
Enable 和 Disable 都调用同一个方法 BuffSystem.OnBuffChanged。这个方法会重新遍历角色身上所有 Buff 的 ActionControlEffect，计算当前的控制状态（是否被眩晕/沉默/禁锢等）。

这是一种声明式的状态管理 — 不关心「加了什么减了什么」，每次变更都从头算一遍「当前总共有什么控制」。简单粗暴但绝对正确。

5.12 Effect → Action 转换模式

Effect 系统中最重要的设计模式是 Effect → Action 的转换。Effect 是「能力声明」（静态配置），Action 是「行为执行」（运行时实例）。

Trigger 触发条件满足
如：技能命中目标、计时器到期
↓
创建 EffectAssignAction
携带 Target + AbilityEffect 引用
↓
AbilityEffectSystem.TriggerApply(effectAssign)
分发到对应的 EffectXxxSystem.TriggerApply
↓
OwnerEntity.XxxAbility.TryMakeAction()
通过工厂 Ability 创建具体 Action 实例
↓
XxxActionSystem.Execute(action)
执行实际效果（扣血/加血/加 Buff/...）

Effect → Action 映射表

Effect 类型组件工厂 AbilityAction 类型ActionSystem
DamageEffect EffectDamageComponent DamageAbility DamageAction DamageActionSystem
CureEffect EffectCureComponent CureAbility CureAction CureActionSystem
AddStatusEffect EffectAddBuffComponent AddStatusAbility AddBuffAction AddBuffActionSystem
AttributeModifyEffect EffectAttributeModifyComponent —（直接操作 Numeric） — —
ActionControlEffect EffectActionControlComponent —（直接通知 BuffSystem） — —
ShieldDefenseEffect EffectShieldDefenseComponent —（直接添加组件） — —
两种触发模式
上表揭示了两种不同的触发模式：

模式 A：通过 Action 执行（Damage/Cure/AddBuff）
Effect → TryMakeAction → Action → ActionSystem.Execute
适用于：一次性执行的行为，需要目标，需要记录战斗日志

模式 B：直接生效（Attribute/ActionControl/Shield）
TriggerApply 时直接操作组件或通知系统
适用于：持续性效果，通过 Enable/Disable 控制生命周期

Attribute 和 ActionControl 使用 Enable/Disable 机制（Buff 激活时生效，结束时回退）。它们不需要 Action 是因为效果是可逆的。

5.13 设计总结

组件即声明
AbilityEffect 挂什么组件就是什么效果。工厂方法 Create 根据配置类型自动挂载。配置驱动 + 组件组合 = 无限可扩展性。添加新效果类型只需：新配置类 + 新 Component + 新 System。
所有 System 结构统一
每个 EffectXxxSystem 都遵循相同模式：
• Awake：缓存配置引用
• Enable/Disable（可选）：持续效果
• TriggerApply（静态）：一次性效果
阅读任何一个就能理解全部。
TriggerApply 的手动分发
AbilityEffectSystem.TriggerApply 用 if (item is XxxComponent) 手动分发到各 System。这不是 EcsNode 的自动驱动，而是手动调度。好处是执行顺序可控；坏处是每加一种新 Effect 就得在这里多加一个 if。
已知局限
• TriggerApply 的 if 链不是开闭原则友好的
• 属性修改的 BaseValue 模式在 Disable 时没有回退逻辑（SetBase 是覆盖操作）
• 装饰器目前只实现了 DamageReduceWithTargetCount 一种，扩展点存在但不完善

决定 Effect 何时、对谁、在什么条件下生效。Trigger 是 Ability 逻辑层的「时序调度器」— 它不做伤害/治疗/加 Buff 的实际操作，只负责判定「此刻是否应该触发效果」，然后把控制权交给 Effect。

6.2 AbilityTriggerSystem.Create — 工厂方法

Trigger 的创建发生在 Ability 初始化阶段。每个配置中的 TriggerConfig 对应创建一个 AbilityTrigger 子实体。

AbilityTriggerSystem.cs — Create
public static AbilityTrigger Create(Ability ownerAbility, TriggerConfig triggerConfig)
{
    // 1. 创建 Trigger 子实体
    var trigger = ownerAbility.AddChild<AbilityTrigger>(x =>
    {
        x.OwnerAbility = ownerAbility;
        x.OwnerEntity = ownerAbility.OwnerEntity;
        x.ParentEntity = ownerAbility.Parent;
        x.TriggerConfig = triggerConfig;
        x.ConditionParamValue = triggerConfig.ConditionParamValue;
    });

    // 2. 解析关联的 Effect 列表
    foreach (var effectId in triggerConfig.TriggerEffectIds)
    {
        trigger.TriggerEffects.Add(ownerAbility.AbilityEffects[effectId]);
    }

    // 3. 如果配置了状态检查条件，添加 TriggerStateCheckComponent
    if (triggerConfig.StateCheckList != null && triggerConfig.StateCheckList.Count > 0)
    {
        trigger.AddComponent<TriggerStateCheckComponent>(comp =>
        {
            comp.StateCheckList = triggerConfig.StateCheckList;
        });
    }

    return trigger;
}
关键设计 — 条件检查的可选性
不是所有 Trigger 都需要状态检查。只有配置了 StateCheckList 的才会挂载 TriggerStateCheckComponent。没有这个组件的 Trigger 在触发时直接跳过状态检查 — 消除特殊情况（好品味原则：组件不存在 = 不需要检查，无需 if/else 判断）。

6.3 AbilityTriggerSystem.Enable — 三种触发模式

当 Ability 被启用时，所有子 Trigger 的 Enable 被级联设为 true，触发 AbilityTriggerSystem.Enable()。根据 AutoTriggerType 配置，启用不同的监听机制。

AbilityTriggerSystem.cs — Enable
public void Enable(AbilityTrigger entity)
{
    var triggerConfig = entity.TriggerConfig;

    // ═══════════ 模式 1：立即触发 ═══════════
    if (triggerConfig.AutoTriggerType == EffectAutoTriggerType.Instant)
    {
        // Buff 生效的瞬间直接执行一次触发
        OnTrigger(entity, new TriggerContext { Target = entity.ParentEntity });
    }
    // ═══════════ 模式 2：行为事件触发 ═══════════
    else if (triggerConfig.AutoTriggerType == EffectAutoTriggerType.Action)
    {
        // 添加行为监听组件 — 订阅 ActionPoint 事件
        entity.AddComponent<BehaviourObserveComponent>(comp =>
        {
            comp.ActionPointType = triggerConfig.ActionPointType;
        });
    }
    // ═══════════ 模式 3：条件/定时触发 ═══════════
    else if (triggerConfig.AutoTriggerType == EffectAutoTriggerType.Condition)
    {
        // 添加时间状态监听组件 — 计时器驱动
        entity.AddComponent<TimeStateObserveComponent>(comp =>
        {
            comp.ConditionParam = triggerConfig.ConditionParam;
            comp.ConditionParamValue = triggerConfig.ConditionParamValue;
        });
    }
}
Instant — 立即触发
Buff 生效瞬间执行一次。典型场景：「装备一件武器，立即获得 +20 攻击力」。Trigger 的存在只是为了走统一的 OnTrigger 流程（状态检查 + Effect 分发）。
Action — 行为事件触发
监听特定行为事件（如 PostReceiveDamage、PostCauseDamage、PreAttack）。典型场景：「受到攻击时，30% 概率反弹 100 伤害」。通过 BehaviourObserveComponent 订阅 ActionPoint。
Condition — 条件/定时触发
基于时间或条件的触发。典型场景：「每 2 秒回复 50 HP」、「3 秒内未受伤则回复 200 HP」。通过 TimeStateObserveComponent 驱动计时器。

6.4 AbilityTriggerSystem.OnTrigger — 核心触发流程

无论哪种触发模式，最终都调用 OnTrigger。这是整个触发系统最关键的方法 — 它连接了「触发条件判定」和「效果执行」两个世界。

AbilityTriggerSystem.cs — OnTrigger（完整流程）
public static void OnTrigger(AbilityTrigger trigger, TriggerContext context)
{
    // ═══════════ Step 1: 目标决议 ═══════════
    // 优先级：context.Target > context.TriggerSource > ParentEntity
    var target = context.Target ?? context.TriggerSource ?? trigger.ParentEntity;

    // ═══════════ Step 2: 状态检查（如果有 TriggerStateCheckComponent）═══════════
    var stateCheckComp = trigger.GetComponent<TriggerStateCheckComponent>();
    if (stateCheckComp != null)
    {
        var passed = TriggerStateCheckSystem.CheckTargetState(
            stateCheckComp, target as CombatEntity);

        if (!passed)
        {
            // 条件未满足，中止触发
            return;
        }
    }

    // ═══════════ Step 3: 遍历所有关联 Effect，创建 EffectAssignAction ═══════════
    foreach (var effect in trigger.TriggerEffects)
    {
        // 跳过未启用的 Effect
        if (!effect.Enable) continue;

        // 创建 EffectAssignAction — 将 Effect 应用到目标
        var assignAction = trigger.OwnerEntity.AddChild<EffectAssignAction>(x =>
        {
            x.AbilityEffect = effect;
            x.Target = target as CombatEntity;
            x.TriggerContext = context;
        });

        // 执行 Action — 这会调用 AssignActionSystem.Execute
        AssignActionSystem.Execute(assignAction);
    }
}
TriggerContext 数据结构
TriggerContext.cs
public class TriggerContext
{
    public EcsEntity Target;          // 明确指定的目标
    public EcsEntity TriggerSource;   // 触发源（如攻击者）
    public Action ActionData;         // 触发时的行动数据（可选）
}
目标决议的优先级逻辑
context.Target ?? context.TriggerSource ?? trigger.ParentEntity

场景解读：
• context.Target 有值：明确目标（如 Execution 时间轴指定的目标）
• 只有 TriggerSource：反击类效果，攻击者自身就是效果目标（如反弹伤害给攻击者）
• 都没有：自身效果，目标是 Buff 持有者（如每秒回血）

这三行代码消除了大量的条件分支。不需要 if (isReflect) ... else if (isSelfBuff) ... else ... 的丑陋结构。
设计洞察 — OnTrigger 是唯一出口
所有触发模式（Instant/Action/Condition）最终都调用同一个 OnTrigger。这意味着：
• 状态检查逻辑只写一次
• Effect 分发逻辑只写一次
• 新增触发模式只需要在 Enable 中添加新的观察组件，不需要改 OnTrigger

这是经典的漏斗模式：多个入口，一个出口。

6.5 BehaviourObserveSystem — 行为事件监听

BehaviourObserveComponent 监听 CombatEntity 上的 ActionPoint 事件。当指定的行为事件发生时（如受到伤害），回调触发 OnTrigger。

BehaviourObserveSystem.cs — 完整源码
public class BehaviourObserveSystem
    : AComponentSystem<AbilityTrigger, BehaviourObserveComponent>,
      IAwake<AbilityTrigger, BehaviourObserveComponent>,
      IDestroy<AbilityTrigger, BehaviourObserveComponent>
{
    // ═══════════ 组件创建时 — 订阅事件 ═══════════
    public void Awake(AbilityTrigger entity, BehaviourObserveComponent component)
    {
        // 解析 ActionPoint 类型（如 "PostReceiveDamage" → ActionPointType 枚举）
        var actionPointType = (ActionPointType)Enum.Parse(
            typeof(ActionPointType), component.ActionPointType);

        // 获取目标实体的 ActionPoint 组件
        var combatEntity = entity.ParentEntity as CombatEntity;
        var actionPoint = combatEntity.GetComponent<ActionPointComponent>();

        // 订阅指定的行为事件
        component.Callback = (triggerContext) =>
        {
            AbilityTriggerSystem.OnTrigger(entity, triggerContext);
        };

        actionPoint.AddListener(actionPointType, component.Callback);
        component.ActionPoint = actionPoint;
        component.ParsedActionPointType = actionPointType;
    }

    // ═══════════ 组件销毁时 — 取消订阅 ═══════════
    public void Destroy(AbilityTrigger entity, BehaviourObserveComponent component)
    {
        // 从 ActionPoint 中移除监听
        component.ActionPoint?.RemoveListener(
            component.ParsedActionPointType, component.Callback);
        component.Callback = null;
        component.ActionPoint = null;
    }
}

6.6 TimeStateObserveSystem — 时间状态监听

TimeStateObserveComponent 用计时器驱动触发。支持两种模式：周期性触发（每 N 秒一次）和条件触发（N 秒内未受伤则触发）。

TimeStateObserveSystem.cs — 完整源码
public class TimeStateObserveSystem
    : AComponentSystem<AbilityTrigger, TimeStateObserveComponent>,
      IAwake<AbilityTrigger, TimeStateObserveComponent>,
      IUpdate<AbilityTrigger, TimeStateObserveComponent>,
      IDestroy<AbilityTrigger, TimeStateObserveComponent>
{
    // ═══════════ 初始化 — 解析计时器类型 ═══════════
    public void Awake(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        // 解析条件类型
        component.ConditionType = (TimeStateConditionType)Enum.Parse(
            typeof(TimeStateConditionType), component.ConditionParam);

        // 解析时间参数
        component.IntervalTime = float.Parse(component.ConditionParamValue);
        component.Timer = 0f;
        component.IsListening = false;

        // 如果是 "N秒内未受伤" 模式，需要监听受伤事件来重置计时器
        if (component.ConditionType == TimeStateConditionType.WhenInTimeNoDamage)
        {
            var combatEntity = entity.ParentEntity as CombatEntity;
            var actionPoint = combatEntity.GetComponent<ActionPointComponent>();

            component.DamageCallback = (ctx) => WhenReceiveDamage(entity, component);
            actionPoint.AddListener(ActionPointType.PostReceiveDamage,
                component.DamageCallback);
            component.ActionPoint = actionPoint;
        }

        // 启动监听
        StartListen(entity, component);
    }

    // ═══════════ 开始监听 ═══════════
    private static void StartListen(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        component.Timer = 0f;
        component.IsListening = true;
    }

    // ═══════════ 每帧更新 — 计时器递增 ═══════════
    public void Update(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        if (!component.IsListening) return;

        component.Timer += Time.deltaTime;

        if (component.Timer >= component.IntervalTime)
        {
            OnTimeState(entity, component);
        }
    }

    // ═══════════ 时间到达 — 触发逻辑 ═══════════
    private static void OnTimeState(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        switch (component.ConditionType)
        {
            // 周期触发：重置计时器，继续循环
            case TimeStateConditionType.WhenIntervalTime:
                component.Timer = 0f;  // 重置，下个周期继续
                AbilityTriggerSystem.OnTrigger(entity,
                    new TriggerContext { Target = entity.ParentEntity });
                break;

            // 一次性触发：停止监听
            case TimeStateConditionType.WhenInTimeNoDamage:
                component.IsListening = false;  // 触发后停止
                AbilityTriggerSystem.OnTrigger(entity,
                    new TriggerContext { Target = entity.ParentEntity });
                break;
        }
    }

    // ═══════════ 受到伤害时 — 重置计时器 ═══════════
    private static void WhenReceiveDamage(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        // 只有 WhenInTimeNoDamage 模式才需要重置
        if (component.ConditionType == TimeStateConditionType.WhenInTimeNoDamage)
        {
            // 重置计时器 — 重新开始计时
            component.Timer = 0f;
            component.IsListening = true;  // 如果之前已触发过，重新启动
        }
    }

    // ═══════════ 销毁时清理 ═══════════
    public void Destroy(AbilityTrigger entity, TimeStateObserveComponent component)
    {
        component.IsListening = false;

        if (component.DamageCallback != null)
        {
            component.ActionPoint?.RemoveListener(
                ActionPointType.PostReceiveDamage, component.DamageCallback);
            component.DamageCallback = null;
        }
    }
}

6.7 TriggerStateCheckSystem — 状态条件检查

TriggerStateCheckSystem 在 OnTrigger 中被调用，负责判断「当前条件是否满足」。支持多条件 AND 逻辑和条件取反。

TriggerStateCheckSystem.cs — 完整源码
public class TriggerStateCheckSystem
    : AComponentSystem<AbilityTrigger, TriggerStateCheckComponent>,
      IAwake<AbilityTrigger, TriggerStateCheckComponent>
{
    // ═══════════ 初始化 — 解析条件字符串，加载检查类 ═══════════
    public void Awake(AbilityTrigger entity, TriggerStateCheckComponent component)
    {
        component.StateChecks = new List<StateCheckEntry>();

        foreach (var config in component.StateCheckList)
        {
            var entry = new StateCheckEntry();
            var checkName = config.StateCheckType;

            // 解析取反前缀 — "!" 表示条件取反
            if (checkName.StartsWith("!"))
            {
                entry.IsInverted = true;
                checkName = checkName.Substring(1);
            }
            else
            {
                entry.IsInverted = false;
            }

            // 动态加载 StateCheck 类
            // 约定：类名 = "StateCheck_" + 配置字符串
            var typeName = $"EGamePlay.Combat.StateCheck_{checkName}";
            var type = Type.GetType(typeName);

            if (type != null)
            {
                entry.Checker = (IStateCheck)Activator.CreateInstance(type);
                entry.Param = config.Param;  // 传递参数（如概率值）
            }
            else
            {
                ConsoleLog.LogError($"StateCheck type not found: {typeName}");
            }

            component.StateChecks.Add(entry);
        }
    }

    // ═══════════ 目标状态检查 — AND 逻辑 ═══════════
    public static bool CheckTargetState(
        TriggerStateCheckComponent component,
        CombatEntity target)
    {
        foreach (var entry in component.StateChecks)
        {
            if (entry.Checker == null) continue;

            // 调用具体的检查实现
            var result = entry.Checker.Check(target, entry.Param);

            // 如果配置了取反，翻转结果
            if (entry.IsInverted) result = !result;

            // AND 逻辑 — 任一条件失败则整体失败
            if (!result) return false;
        }

        return true;  // 所有条件通过
    }
}

设计洞察 — 取反前缀的优雅性
不需要为每个 StateCheck 写两个版本（如 HasBuff / NotHasBuff）。配置中写 "!HasBuff" 就是取反。

这是 Linus 说的「好品味」— 用一个通用机制（前缀解析 + 结果取反）消除了一半的 StateCheck 类。系统的表达力翻倍，代码量不变。

Chapter 07

AbilityExecution — 执行时间轴

技能的视觉表现层。AbilityExecution 是一个时间轴驱动的执行器 — 它编排动画、粒子、音效、弹道和伤害判定的精确时序。策划在 ScriptableObject 中编辑时间轴，程序在运行时逐帧推进。

7.5 AbilityExecutionSystem — 执行系统

AbilityExecutionSystem 管理 Execution 的生命周期：创建 → 启动时间轴 → 每帧推进 Clip → 结束销毁。

AbilityExecutionSystem.cs — 核心逻辑
public class AbilityExecutionSystem : AEntitySystem<AbilityExecution>,
    IAwake<AbilityExecution>,
    IUpdate<AbilityExecution>,
    IDestroy<AbilityExecution>
{
    // ═══════════ 创建 ═══════════
    public void Awake(AbilityExecution entity)
    {
        var execObj = entity.ExecutionObject;

        // 从 SO 拷贝 Clip 数据到运行时实例
        entity.TotalTime = execObj.TotalTime;
        entity.CurrentTime = 0f;

        foreach (var clipData in execObj.ExecuteClips)
        {
            var runtimeClip = CloneClipData(clipData);
            runtimeClip.HasStarted = false;
            runtimeClip.HasEnded = false;
            entity.ExecuteClips.Add(runtimeClip);
        }

        // 实例化视觉对象（如果有）
        if (execObj.ObjAsset != null)
        {
            // Instantiate 主体预制件
        }
    }

    // ═══════════ 每帧推进时间轴 ═══════════
    public void Update(AbilityExecution entity)
    {
        entity.CurrentTime += Time.deltaTime;

        // 遍历所有 Clip，检查是否该触发
        foreach (var clip in entity.ExecuteClips)
        {
            // Clip 尚未开始 且 当前时间已到达 StartTime
            if (!clip.HasStarted && entity.CurrentTime >= clip.StartTime)
            {
                clip.HasStarted = true;
                OnClipStart(entity, clip);
            }

            // Clip 已开始、尚未结束 且 当前时间已到达 EndTime
            if (clip.HasStarted && !clip.HasEnded && entity.CurrentTime >= clip.EndTime)
            {
                clip.HasEnded = true;
                OnClipEnd(entity, clip);
            }

            // Clip 正在活跃期间（已开始、未结束）— 持续更新
            if (clip.HasStarted && !clip.HasEnded)
            {
                OnClipUpdate(entity, clip);
            }
        }

        // 时间轴结束 → 销毁 Execution
        if (entity.CurrentTime >= entity.TotalTime)
        {
            entity.Parent.RemoveChild(entity);
        }
    }

    // ═══════════ Clip 开始 ═══════════
    private void OnClipStart(AbilityExecution entity, ExecuteClipData clip)
    {
        switch (clip.ExecuteClipType)
        {
            case ExecuteClipType.Animation:
                // 播放动画：设置 Animator 参数
                PlayAnimation(entity, clip.AnimationData);
                break;

            case ExecuteClipType.Audio:
                // 播放音效
                PlayAudio(clip.AudioData);
                break;

            case ExecuteClipType.ParticleEffect:
                // 实例化粒子特效
                SpawnParticle(entity, clip.ParticleEffectData);
                break;

            case ExecuteClipType.ActionEvent:
                // 如果是 StartTrigger 类型，立即触发效果
                if (clip.ActionEventData.FireType == FireType.StartTrigger)
                {
                    FireActionEvent(entity, clip.ActionEventData);
                }
                // CollisionTrigger 类型 — 创建碰撞检测区域
                else
                {
                    CreateCollisionTrigger(entity, clip);
                }
                break;

            case ExecuteClipType.ItemExecute:
                // 生成弹道物体
                SpawnItem(entity, clip.ItemData);
                break;
        }
    }

    // ═══════════ Clip 结束 ═══════════
    private void OnClipEnd(AbilityExecution entity, ExecuteClipData clip)
    {
        switch (clip.ExecuteClipType)
        {
            case ExecuteClipType.ActionEvent:
                if (clip.ActionEventData.FireType == FireType.EndTrigger)
                {
                    FireActionEvent(entity, clip.ActionEventData);
                }
                // 销毁碰撞检测区域
                DestroyCollisionTrigger(clip);
                break;

            case ExecuteClipType.ParticleEffect:
                // 停止粒子发射（让已有粒子自然消亡）
                break;
        }
    }

    // ═══════════ Clip 持续更新（活跃期间每帧调用）═══════════
    private void OnClipUpdate(AbilityExecution entity, ExecuteClipData clip)
    {
        if (clip.ExecuteClipType == ExecuteClipType.ActionEvent)
        {
            if (clip.ActionEventData.FireType == FireType.CollisionTriggerMultiple)
            {
                // 持续碰撞检测 — 每帧检查新进入的目标
                UpdateCollisionTrigger(entity, clip);
            }
        }
    }

    // ═══════════ 销毁 ═══════════
    public void Destroy(AbilityExecution entity)
    {
        // 清理所有视觉对象
        // 释放 ActionOccupy（解除施法锁定）
    }
}
设计洞察 — 时间驱动而非事件驱动
Execution 的核心循环极其简单：每帧检查 CurrentTime 是否到达各 Clip 的 StartTime/EndTime。

不用协程、不用 Invoke、不用事件链。一个 Update 里一个 foreach，清晰透明。
想知道技能第 0.5 秒发生了什么？看 Clip 列表中 StartTime ≤ 0.5 ≤ EndTime 的那些 Clip 就行了。

调试友好度极高 — 暂停游戏后看 CurrentTime 和各 Clip 的 HasStarted/HasEnded 状态就能完全还原执行进度。

7.7 弹道系统 — ItemExecute

当 ItemExecute Clip 开始时，系统生成一个弹道物体实体。这个实体有自己的运动组件和碰撞检测，独立于 Execution 时间轴生存。

弹道物体生成流程
private void SpawnItem(AbilityExecution execution, ItemData itemData)
{
    // 1. 实例化弹道物体 GO
    var itemObj = Object.Instantiate(itemData.ObjAsset);
    itemObj.transform.position = execution.OwnerEntity.Position;

    // 2. 创建弹道实体
    var itemEntity = execution.AddChild<ExecuteItemEntity>(x =>
    {
        x.ItemData = itemData;
        x.OwnerExecution = execution;
        x.GameObject = itemObj;
        x.Speed = itemData.Speed;
        x.Duration = itemData.Duration;
    });

    // 3. 根据 CollisionMoveType 设置运动行为
    switch (itemData.CollisionMoveType)
    {
        case CollisionMoveType.FixedPosition:
            // 放置在 InputPoint 位置，不移动
            itemObj.transform.position = execution.InputPoint;
            break;

        case CollisionMoveType.FollowOwner:
            // 每帧同步到施法者位置
            itemEntity.AddComponent<FollowOwnerComponent>();
            break;

        case CollisionMoveType.FollowTarget:
            // 追踪 InputTarget，到达后触发碰撞
            itemEntity.AddComponent<FollowTargetComponent>(comp =>
            {
                comp.Target = execution.InputTarget;
            });
            break;

        case CollisionMoveType.SelectedDirection:
            // 朝 InputDirection 方向匀速飞行
            itemEntity.AddComponent<DirectionMoveComponent>(comp =>
            {
                comp.Direction = execution.InputDirection;
                comp.Speed = itemData.Speed;
            });
            break;

        case CollisionMoveType.PathFly:
            // 沿路径点列表飞行
            itemEntity.AddComponent<PathFlyComponent>();
            break;

        case CollisionMoveType.Bezier:
            // 贝塞尔曲线飞行（起点 → 控制点 → 目标）
            itemEntity.AddComponent<BezierMoveComponent>(comp =>
            {
                comp.Target = execution.InputTarget;
                comp.ControlPointHeight = 3f;  // 弧线高度
            });
            break;
    }

    // 4. 添加碰撞检测组件
    if (itemData.CollisionTrigger != null)
    {
        itemEntity.AddComponent<CollisionDetectComponent>(comp =>
        {
            comp.TriggerData = itemData.CollisionTrigger;
            comp.OnCollision = (target) => OnItemCollision(execution, itemEntity, target);
        });
    }
}
各 CollisionMoveType 运动行为详解
FixedPosition
弹道物体生成后固定在 InputPoint 位置。持续存在直到 Duration 到期。

用途：AOE 区域技能（如燃烧地面、毒圈）。碰撞检测持续运行，每帧检查范围内的目标。
FollowTarget
每帧向 InputTarget 位置移动。到达目标后触发碰撞回调并自动销毁。

用途：追踪弹道（如阿狸 W 狐火）。目标无法通过移动闪避。速度决定追踪延迟。
SelectedDirection
朝 InputDirection 方向以 Speed 匀速飞行。碰到第一个目标后触发碰撞（或飞到 Duration 上限销毁）。

用途：方向性弹道（如 EZ Q）。可以被闪避。
Bezier
沿贝塞尔曲线从施法者飞向目标。曲线的高度/弧度由控制点参数决定。

用途：弧线弹道（如锤石 Q、高抛技能）。视觉上更有冲击力。
弹道命中回调
private static void OnItemCollision(
    AbilityExecution execution,
    ExecuteItemEntity item,
    CombatEntity target)
{
    // 命中目标后执行 ActionEvent 效果
    var actionEventData = item.ItemData.ActionEventData;
    if (actionEventData != null)
    {
        FireActionEvent(execution, actionEventData, target);
    }

    // 生成命中粒子特效
    if (item.ItemData.HitParticle != null)
    {
        Object.Instantiate(item.ItemData.HitParticle, target.Position, Quaternion.identity);
    }

    // 弹道消亡（单体弹道命中后销毁）
    item.Parent.RemoveChild(item);
}

7.8 ActionEventData — 视觉与逻辑的桥梁

ActionEventData 是 Execution 时间轴中最重要的 Clip 类型 — 它是视觉表现层和战斗逻辑层的唯一连接点。

核心职责
ActionEvent 回答一个关键问题：「在时间轴的某个精确时刻，对谁执行什么效果？」

它把视觉时间轴（动画播到第几帧）映射到逻辑操作（触发 AbilityTrigger → 执行 Effect → 创建 Action）。
FireActionEvent 执行流程
FireActionEvent — 从时间轴到效果系统
private static void FireActionEvent(
    AbilityExecution execution,
    ActionEventData actionEvent,
    CombatEntity collisionTarget = null)
{
    // Step 1: 确定目标
    CombatEntity target = null;
    switch (actionEvent.EffectApplyTarget)
    {
        case EffectApplyTarget.Target:
            target = collisionTarget ?? execution.InputTarget;
            break;
        case EffectApplyTarget.Self:
            target = execution.OwnerEntity;
            break;
        case EffectApplyTarget.SelfAndTarget:
            // 先对自身执行，再对目标执行
            ApplyEffect(execution, execution.OwnerEntity);
            target = collisionTarget ?? execution.InputTarget;
            break;
    }

    if (target == null) return;

    // Step 2: 根据 ActionEventType 选择执行方式
    switch (actionEvent.ActionEventType)
    {
        case ActionEventType.AssignEffect:
            // 走 Trigger → Effect → Action 标准流程
            ApplyEffect(execution, target);
            break;

        case ActionEventType.TriggerExecution:
            // 触发另一个 Execution（连锁技能/子技能）
            TriggerSubExecution(execution, target, actionEvent);
            break;
    }
}

private static void ApplyEffect(AbilityExecution execution, CombatEntity target)
{
    // 获取关联的 Ability 的 Trigger
    var ability = execution.AbilityEntity;

    // 对每个 Trigger 调用 OnTrigger — 传入碰撞目标作为 Context.Target
    foreach (var trigger in ability.AbilityTriggers)
    {
        AbilityTriggerSystem.OnTrigger(trigger,
            new TriggerContext { Target = target });
    }
}

FireType何时用碰撞检测典型示例

Execution 完整生命周期
1. 玩家按 Q → 创建 AbilityExecution 实体
2. Awake: 拷贝 SO 数据、实例化视觉对象
3. Update: 每帧推进 CurrentTime，触发到时的 Clip
4. Clip 触发: 播动画/音效/粒子/生成弹道/判定伤害
5. TotalTime 到达 → RemoveChild → Destroy
6. 清理视觉对象、释放行为占用
与 Ability 的关系
• Ability 是定义（一直存在）
• Execution 是执行（临时存在）
• 每次释放技能 = 创建一个新 Execution
• CD 管理在 Ability 层，不在 Execution 层
• Execution 不关心 CD — 它只关心「被创建后如何执行」
已知限制
• 不支持时间轴倒放/暂停/变速（只能匀速前进）
• Clip 不能跨 TotalTime 边界（EndTime 必须 ≤ TotalTime）
• 弹道物体的生命周期独立于 Execution — Execution 销毁后弹道仍在飞
• 无法在运行时动态增删 Clip（SO 是只读的）
性能注意
• 每帧遍历所有 Clip 的 O(N) 开销在 Clip 少时可忽略
• 弹道物体多时（如乱战中大量飞弹）注意 GameObject 实例化开销
• CollisionTriggerMultiple 每帧做碰撞检测 — 范围越大开销越高
• 考虑为弹道物体加入对象池（当前未实现）

Chapter 08

Action — 行动系统

战斗中每一次「发生的事」都是一个 Action：释放技能、造成伤害、治疗、施加 Buff、普攻、碰撞、效果分配。Action 系统通过 8 阶段管线将每个行动拆解为可插入、可拦截、可修改的独立步骤，是整个战斗流程的脊柱。

8.1 概述

核心定位
Action 系统是 EGamePlay 的行动管线。它解决的核心问题：
• 如何统一处理「释放技能」「造成伤害」「治疗」「加 Buff」等不同行为？
• 如何在行动执行的各个阶段插入逻辑（如护甲减伤、暴击判定、伤害反弹）？
• 如何让攻击方和受击方都能在各自的时机做出响应？

答案：每个行为都是一个 EcsEntity，由工厂创建，经过 8 阶段管线处理后销毁。

8.2 IActionAbility — 工厂接口

CombatEntity 持有多个 IActionAbility 实现，用于创建对应类型的 Action。这是工厂模式 — 每种行动类型有自己的工厂。

IActionAbility.cs
namespace EGamePlay.Combat
{
    /// Action 工厂接口
    /// CombatEntity 通过此接口创建各种 Action 实体
    public interface IActionAbility
    {
        /// 尝试创建一个 Action 实例
        /// 返回 false 表示当前条件不满足（如 CD 中、被沉默等）
        bool TryMakeAction(out IActionExecute action);
    }
}
工厂模式的意义
TryMakeAction 返回 bool 允许在创建阶段就进行条件检查：
• SpellAction 工厂：检查 CD、蓝量、是否被沉默
• AttackAction 工厂：检查是否被缴械
• DamageAction 工厂：通常总是 true（伤害由系统内部触发）

这样上层调用方只需要 if (ability.TryMakeAction(out var action))，不需要关心具体的前置条件。

• 伤害统计面板（技能 X 造成了多少伤害）
• 条件触发（当某个特定技能造成暴击时…）
• 调试日志（为什么这个伤害这么高？追溯源头）

    /// 当触发器决定「现在应该执行某个效果」时，创建此 Action
    /// AssignActionSystem 再根据效果类型派发到具体的 Action
    public class EffectAssignAction : EcsEntity, IActionExecute
    {
        public ActionType ActionType => ActionType.EffectAssignAction;

        // ═══════════ IActionExecute ═══════════
        public CombatEntity Creator { get; set; }
        public CombatEntity Target { get; set; }
        public EffectAssignAction SourceAssignAction { get; set; }
        public IActionAbility ActionAbility { get; set; }

        // ═══════════ EffectAssignAction 特有数据 ═══════════
        public AbilityEffect AbilityEffect { get; set; }      // 要执行的效果实体
        public CombatEntity AssignTarget { get; set; }        // 效果分配目标
        public Ability SourceAbility { get; set; }             // 来源技能
        public TriggerContext TriggerContext { get; set; }     // 触发上下文
    }
}
EffectAssignAction 是调度枢纽
EffectAssignAction 是触发器系统（Ch06）和行动系统的桥梁：
• 触发器检测到条件满足 → 创建 EffectAssignAction
• AssignActionSystem 读取 AbilityEffect 的类型（伤害/治疗/加Buff/...）
• 派发到对应的具体 System（DamageActionSystem / CureActionSystem / ...）

这实现了「效果类型」和「执行逻辑」的解耦 — 添加新效果类型只需新增一个分支。

8.5 ActionSystem — 8 阶段生命周期

所有 Action 执行时都经过统一的 8 阶段管线。每个阶段是一个扩展点，可以通过挂载对应的 ActionPoint 来插入逻辑（如护盾吸收、伤害反弹、暴击加成）。

阶段总览

Phase 1: BeforeExecuteAction
攻击方 — 行动前置检查/修改
↓
Phase 2: BeforeSufferAction
受击方 — 受击前置检查/修改
↓
Phase 3: ActionCheckProcess
验证 + 核心计算（伤害值/治疗值）
↓
Phase 4: BeforeCauseApply
攻击方 — 应用前修改（最后的加成）
↓
Phase 5: BeforeReceiveApply
受击方 — 应用前修改（护盾/减伤）
↓
Phase 6: ActionApply
执行！扣血/加血/施加Buff
↓
Phase 7: AfterCauseApply + AfterReceiveApply
双方事后处理（吸血/反伤/触发被动）
↓
Phase 8: AfterExecuteAction + AfterSufferAction
双方最终清理 + 统计

完整管线源码

ActionSystem.cs — Execute 方法
public static class ActionSystem
{
    /// 执行一个 Action 的完整 8 阶段管线
    public static void Execute(IActionExecute action)
    {
        var creator = action.Creator;
        var target = action.Target;

        // ═══════════ Phase 1: BeforeExecuteAction ═══════════
        // 攻击方的前置处理点
        // 用途：攻击方的 Buff 可以在此修改行动（如「下次伤害翻倍」）
        creator.TriggerActionPoint(ActionPointType.BeforeExecuteAction, action);

        // ═══════════ Phase 2: BeforeSufferAction ═══════════
        // 受击方的前置处理点
        // 用途：受击方的 Buff 可以在此拦截（如「无敌状态免疫伤害」）
        target.TriggerActionPoint(ActionPointType.BeforeSufferAction, action);

        // ═══════════ Phase 3: ActionCheckProcess ═══════════
        // 核心计算阶段
        // 由各个具体的 ActionSystem 实现（如 DamageActionSystem 计算伤害）
        ActionCheckProcess(action);

        // ═══════════ Phase 4: BeforeCauseApply ═══════════
        // 攻击方的应用前处理
        // 用途：「造成伤害前，额外附加 X% 真实伤害」
        creator.TriggerActionPoint(ActionPointType.BeforeCauseApply, action);

        // ═══════════ Phase 5: BeforeReceiveApply ═══════════
        // 受击方的应用前处理
        // 用途：护盾吸收、伤害减免百分比
        target.TriggerActionPoint(ActionPointType.BeforeReceiveApply, action);

        // ═══════════ Phase 6: ActionApply ═══════════
        // 真正的执行 — 修改数值/状态
        // 伤害：target.HP -= damage
        // 治疗：target.HP += cure
        // 加 Buff：target.AddAbility(buff)
        ActionApply(action);

        // ═══════════ Phase 7: AfterCauseApply + AfterReceiveApply ═══════════
        // 双方的事后处理
        // 攻击方用途：生命偷取（造成伤害后回血）
        // 受击方用途：荆棘护甲（受到伤害后反弹）
        creator.TriggerActionPoint(ActionPointType.AfterCauseApply, action);
        target.TriggerActionPoint(ActionPointType.AfterReceiveApply, action);

        // ═══════════ Phase 8: AfterExecuteAction + AfterSufferAction ═══════════
        // 最终阶段 — 通常用于统计和 UI 更新
        // 攻击方：记录「本次造成了 X 伤害」
        // 受击方：触发飘字、受击动画、死亡检测
        creator.TriggerActionPoint(ActionPointType.AfterExecuteAction, action);
        target.TriggerActionPoint(ActionPointType.AfterSufferAction, action);
    }
}

各阶段的 MOBA 应用场景

阶段角色MOBA 实例
Phase 1 BeforeExecuteAction攻击方露娜被动：技能后下一次普攻强化
Phase 2 BeforeSufferAction受击方张飞大招：免疫控制效果
Phase 3 ActionCheckProcess系统计算 ATK×(ATK/(ATK+DEF)) 基础伤害
Phase 4 BeforeCauseApply攻击方末世：额外造成目标当前HP 8% 伤害
Phase 5 BeforeReceiveApply受击方不祥征兆：减少 35% 物理伤害
Phase 6 ActionApply系统实际扣血 target.HP -= finalDamage
Phase 7 AfterCause/ReceiveApply双方吸血书(攻方回血) / 反甲(受方反伤)
Phase 8 AfterExecute/SufferAction双方更新 UI 飘字、击杀判定、助攻记录
设计洞察 — 为什么是 8 阶段而不是更简单的 Before/After
关键在于攻守双方各有独立的插入点：
• Phase 1 vs Phase 2：攻击方增伤 和 受击方减免 是独立的
• Phase 4 vs Phase 5：应用前的最终修改也是双方独立的
• Phase 7：事后处理也是双方独立的（吸血 vs 反伤）

如果只有 Before/After，你无法区分「谁的 Buff 在什么时候生效」。8 阶段让优先级完全明确。

8.6 DamageActionSystem.Execute — 伤害计算

伤害计算是 ActionCheckProcess 阶段的具体实现。它根据伤害来源（普攻/技能/Buff）走不同的计算路径。

DamageActionSystem.cs — 完整执行逻辑
public class DamageActionSystem : AEntitySystem<DamageAction>, IAwake<DamageAction>
{
    public void Awake(DamageAction action)
    {
        // DamageAction 创建后立即执行管线
        Execute(action);
    }

    private void Execute(DamageAction action)
    {
        var creator = action.Creator;
        var target = action.Target;

        // ═══════════ 获取攻防属性 ═══════════
        var atk = creator.GetComponent<AttributeComponent>().Attack.Value;
        var def = target.GetComponent<AttributeComponent>().Defense.Value;

        // ═══════════ 基础伤害计算 ═══════════
        int damage = 0;

        if (action.DamageSource == DamageSource.Attack)
        {
            // 普攻伤害路径：纯靠攻防公式
            damage = (int)(atk * (atk / (atk + def)));
        }
        else if (action.DamageSource == DamageSource.Skill)
        {
            // 技能伤害路径：读取效果配置的伤害公式
            var abilityEffect = action.SourceAssignAction.AbilityEffect;
            damage = EffectDamageSystem.GetDamageValue(abilityEffect);
            // ↑ 解析 DamageValueFormula 表达式，替换运行时参数
        }
        else if (action.DamageSource == DamageSource.Buff)
        {
            // Buff/DOT 伤害路径：同技能
            var abilityEffect = action.SourceAssignAction.AbilityEffect;
            damage = EffectDamageSystem.GetDamageValue(abilityEffect);
        }

        // ═══════════ 暴击判定 ═══════════
        var critProb = creator.GetComponent<AttributeComponent>()
            .CriticalProbability.Value;
        if (Random.Range(0f, 1f) < critProb / 100f)
        {
            damage = (int)(damage * 1.5f);
            action.IsCritical = true;
        }

        // ═══════════ 记录最终伤害值 ═══════════
        action.DamageValue = damage;

        // ═══════════ 走 8 阶段管线 ═══════════
        ActionSystem.Execute(action);

        // ═══════════ Phase 6 ActionApply: 扣血 ═══════════
        // （在 ActionApply 阶段执行）
        var hp = target.GetComponent<AttributeComponent>().HealthPoint;
        var newHp = hp.Value - action.DamageValue;
        hp.SetBase(newHp);
        // ↑ 直接修改 baseValue，因为 HP 的当前值就是 base
        // （HP 的 Modifier 层用于最大生命值加成，当前值用 base 表示）
    }
}
伤害公式解析
核心公式：damage = atk × (atk / (atk + def))

这是一个递减收益防御公式：
• atk=100, def=0 → damage = 100×(100/100) = 100（无防御 = 全额伤害）
• atk=100, def=100 → damage = 100×(100/200) = 50（等防 = 半伤）
• atk=100, def=300 → damage = 100×(100/400) = 25（3倍防 = 25%伤害）
• atk=100, def=∞ → damage → 0（但永远不为 0）

特性：防御越堆越不值（递减收益），攻击永远能打出伤害（不会被 100% 减免）。
暴击 1.5 倍 — 写死还是配置化？
当前暴击倍率 1.5f 是硬编码的。在正式项目中应该：
• 抽成 AttributeComponent.CriticalDamageMultiplier（也是 FloatNumeric）
• 这样「暴击伤害提升」的装备/Buff 可以通过 Modifier 系统修改它
• 当前实现是 Demo 简化，后续扩展不需要改架构

8.7 SpellActionSystem.Execute — 施法执行

SpellAction 的核心职责：创建 AbilityExecution（执行体），注入输入参数，启动时间轴。

SpellActionSystem.cs — 完整执行逻辑
public class SpellActionSystem : AEntitySystem<SpellAction>, IAwake<SpellAction>
{
    public void Awake(SpellAction action)
    {
        Execute(action);
    }

    private void Execute(SpellAction action)
    {
        var creator = action.Creator;
        var ability = action.ActionAbility as Ability;

        // ═══════════ 1. 创建执行体 ═══════════
        // AbilityExecution 是技能的「运行时实例」
        // 一个 Ability 可以同时有多个 Execution（如持续施法/蓄力）
        var execution = creator.AddChild<AbilityExecution>(x =>
        {
            x.Ability = ability;
            x.ExecutionObject = ability.ConfigObject.ExecutionObject;
        });

        // ═══════════ 2. 注入输入参数 ═══════════
        // 这些值来自玩家的输入（点击目标/点击地面/滑动方向）
        execution.InputTarget = action.InputTarget;
        execution.InputPoint = action.InputPoint;
        execution.InputDirection = action.InputDirection;

        // ═══════════ 3. 保存引用 ═══════════
        action.AbilityExecution = execution;

        // ═══════════ 4. 启动执行时间轴 ═══════════
        // 触发 AbilityExecutionSystem.Awake → 加载 ExecutionObject
        // → 解析 Timeline Clips → 开始逐帧播放
        execution.Init();
        // ↑ Init 触发 IInit System，开始运行时间轴
        // 时间轴中的 Clip 会在对应时间点触发 EffectAssignAction
        // → 再触发 DamageAction / AddBuffAction 等
    }
}
SpellAction 是「起点」
SpellAction 不直接造成伤害或效果 — 它只负责创建和启动 AbilityExecution。
后续的伤害/Buff 由时间轴中的 Clip 在正确的时间点触发。

完整链路：
SpellAction → AbilityExecution.Init() → Timeline 播放 → Clip 触发 → EffectAssignAction → DamageAction/AddBuffAction

8.8 AddBuffActionSystem.Execute — 加 Buff

AddBuffAction 负责将一个 Buff（本质上是 Ability 实体）挂载到目标身上。

AddBuffActionSystem.cs — 完整执行逻辑
public class AddBuffActionSystem : AEntitySystem<AddBuffAction>, IAwake<AddBuffAction>
{
    public void Awake(AddBuffAction action)
    {
        Execute(action);
    }

    private void Execute(AddBuffAction action)
    {
        var target = action.Target;
        var addStatusEffect = action.AddStatusEffect;

        // ═══════════ 1. 获取 Buff 配置 ═══════════
        // AddStatusEffect 持有要施加的 Buff 的 ScriptableObject 引用
        var buffConfigObject = addStatusEffect.StatusConfigObject;

        // ═══════════ 2. 挂载 Buff 到目标 ═══════════
        // AbilitySystem.Attach 创建 Ability 子实体
        // Buff 在 EGamePlay 中就是 Type="Buff" 的 Ability
        var buff = AbilitySystem.Attach(target, buffConfigObject);

        // ═══════════ 3. 设置持续时间 ═══════════
        // 从 AddStatusEffect 配置读取持续时间
        var durationComponent = buff.GetComponent<AbilityDurationComponent>();
        if (durationComponent != null && addStatusEffect.Duration > 0)
        {
            durationComponent.Duration = addStatusEffect.Duration;
            durationComponent.GameTimer.Reset();
        }

        // ═══════════ 4. 参数替换 ═══════════
        // Buff 的效果参数可能引用施法者的属性
        // 如「造成施法者攻击力 * 50% 的持续伤害」
        // 需要在挂载时将施法者属性快照到 Buff 参数中
        if (action.SourceAbility != null)
        {
            var sourceParams = action.SourceAbility.ConfigObject.Parameters;
            if (sourceParams != null)
            {
                foreach (var param in sourceParams)
                {
                    // 将源技能的参数传递到 Buff 的效果中
                    // 如源技能的攻击力倍率 → Buff 的 DOT 伤害公式参数
                    buff.SetParameter(param.Key, param.Value);
                }
            }
        }

        // ═══════════ 5. 激活 Buff ═══════════
        buff.Enable = true;
        // ↑ 触发 IEnable → 所有 Effect 的 Component System 启动
        // → 属性修改器开始生效 / 触发器开始监听
    }
}
Buff = Ability 的统一模型
在 EGamePlay 中 Buff 不是独立系统 — 它就是一个 Type="Buff" 的 Ability：
• 同样有 Effect 子实体（属性修改/DOT伤害/控制效果）
• 同样通过 AbilitySystem.Attach 挂载
• 同样用 Enable/Disable 控制激活状态
• 结束时销毁 → 所有 Modifier 自动 Remove → 数值恢复

这是整个框架最优雅的设计之一 — Skill 和 Buff 共享完整的技能管线。

8.9 AssignActionSystem.Execute — 效果分配调度

AssignActionSystem 是触发器系统和行动系统的桥梁。它根据 AbilityEffect 的类型，派发到对应的具体 Action。

AssignActionSystem.cs — 调度逻辑
public class AssignActionSystem : AEntitySystem<EffectAssignAction>, IAwake<EffectAssignAction>
{
    public void Awake(EffectAssignAction action)
    {
        Execute(action);
    }

    private void Execute(EffectAssignAction action)

        // ═══════════ 根据效果类型派发 ═══════════

        // 这个方法检查效果实体上挂了什么 Component，然后触发对应逻辑

        // TriggerApply 内部的分发逻辑：
        //
        // if (effect.HasComponent<EffectDamageComponent>())

        //

        //     → 创建 CureAction
        //
        // if (effect.HasComponent<EffectAddBuffComponent>())
        //     → 创建 AddBuffAction
        //
        // if (effect.HasComponent<EffectAttributeModifyComponent>())
        //     → 直接操作 Modifier（不需要独立 Action）
        //
        // if (effect.HasComponent<EffectActionControlComponent>())
        //     → 施加行动控制（眩晕/沉默/缴械）
    }
}
数据流全景

8.10 设计总结

Action = 短命 Entity
每个 Action 是 EcsEntity 实例，携带该行为的全部上下文。创建 → 管线 → 销毁。生命周期通常在一帧内完成。这意味着每次伤害都有独立的 DamageValue、IsCritical 等数据，不会互相污染。
8 阶段 = 完美扩展点
新需求（如「伤害反弹」「吸血」「护盾」）不需要修改 ActionSystem 核心代码 — 只需要在对应的 ActionPoint 注册回调。开闭原则的教科书实现。
EffectAssignAction 是枢纽
触发器系统（Ch06）通过 EffectAssignAction 连接到行动系统。它是「什么时候执行」和「执行什么」之间的解耦点。去掉它，触发器就要直接知道行动系统的内部细节。
已知局限
• 每次行动 new 一个 Entity，GC 压力（无对象池）
• 暴击倍率硬编码 1.5f
• 管线无法中途打断（如闪避应该跳过 Phase 6，当前需要在 Phase 5 将 damage 设为 0）
• 无异步支持（弹道飞行时间怎么办？答案在 Ch07 Execution）

Chapter 09

Configuration Pipeline — 配置管线

策划如何配置一个技能？从 Excel 表格到运行时数据结构，从 ScriptableObject 到时间轴编辑器。本章解析 EGamePlay 的双轨配置架构：Excel 管参数，ScriptableObject 管结构。

9.1 概述

核心问题
一个 MOBA 技能的配置信息包含：
• 参数型数据：CD、伤害值、范围、持续时间 → 策划频繁调整
• 结构型数据：几个效果？什么触发条件？时间轴如何编排？ → 设计阶段确定

用单一方案管理两类数据会出问题：
• 纯 Excel → 无法表达嵌套结构（效果列表、触发条件树）
• 纯 ScriptableObject → 策划改个 CD 要打开 Unity、找资源、改值、保存

解决方案：双轨。

9.2 双轨架构

Excel — 参数轨
管什么：数值参数
工具：Excel → 导出工具 → JSON → C# 类
谁用：策划日常调参
频率：每天多次
产物：AbilityConfig 类（运行时配置）
ScriptableObject — 结构轨
管什么：效果组合、触发逻辑、时间轴
工具：Unity Inspector + 自定义编辑器
谁用：技能设计师
频率：设计阶段
产物：AbilityConfigObject + ExecutionObject

两轨如何关联

AbilityConfig (Excel)
Id = 1001, Name = "火球术", CD = 8s
↓ Id 关联
AbilityConfigObject (SO)
Id = 1001, Effects = [DamageEffect, AddStatusEffect]
↓ ExecutionObject 引用
ExecutionObject (SO)
Timeline: 0.3s 播特效, 0.5s 发弹道, 0.8s 碰撞检测
Id 是两轨的桥梁
AbilityConfig.Id 和 AbilityConfigObject.Id 相同时，运行时自动关联。
策划在 Excel 改 CD = 6s → 导出 → 运行时 AbilityConfig 更新 → Ability 读到新 CD。
不需要碰 ScriptableObject，不需要打开 Unity。

AbilityConfigObject 是 Unity ScriptableObject，在 Inspector 中可视化编辑。它定义技能/Buff 的效果组合和触发逻辑。

namespace EGamePlay.Combat
{
    /// 技能的目标影响类型
    public enum AffectTargetType
    {
        SelfTeam,       // 己方
        EnemyTeam,      // 敌方
        Self,           // 仅自身
    }

    /// 目标选取类型
    public enum TargetSelectType
    {
        CollisionSelect,    // 碰撞检测
        ConditionSelect,    // 条件指定

    /// 技能/Buff 的结构配置（ScriptableObject）

        public AffectTargetType AffectTargetType;
        public TargetSelectType TargetSelectType;

        // ═══════════ 效果列表 ═══════════
        // 一个技能可以有多个效果（如：造成伤害 + 施加减速 + 回血）
        [SerializeReference]

[SerializeReference] 的魔力
Effects 列表用 [SerializeReference] 而不是 [SerializeField]：
• 允许列表中存放 Effect 的不同子类（DamageEffect、CureEffect、...）
• Unity Inspector 会显示每个效果的具体类型字段
• 这实现了「一个技能多个不同类型的效果」的组合配置

9.6 TriggerConfig — 触发配置

TriggerConfig 定义效果在什么条件下执行。每个 TriggerConfig 关联一组 Effect，描述「什么时候」执行「哪些效果」。

TriggerConfig.cs — 完整结构
namespace EGamePlay.Combat
{
    /// 触发方式
    public enum TriggerType
    {
        AutoTrigger,    // 自动触发（由系统驱动）
        ManualTrigger,  // 手动触发（由时间轴 Clip 驱动）
    }

    /// 自动触发的子类型
    public enum AutoTriggerType
    {
        Instant,    // 立即（Ability 激活时）
        Action,     // 行动点（某个 ActionPoint 触发时）
        Condition,  // 条件（时间/状态/事件满足时）
    }

    [Serializable]
    public class TriggerConfig
    {
        // ═══════════ 触发方式 ═══════════
        public TriggerType TriggerType;

        // ═══════════ 自动触发配置 ═══════════
        public AutoTriggerType AutoTriggerType;

        /// Action 触发时：在哪个行动点触发
        /// 如 AfterReceiveApply = 受到伤害后触发
        public ActionPointType ActionPointType;

        /// Condition 触发时：条件类型
        /// 如 WhenHPBelowPercent / WhenTimeInterval
        public TimeStateEventType ConditionType;

        // ═══════════ 条件参数 ═══════════
        /// 状态检查列表
        /// 如 ["HP < 30%", "HasBuff_Shield == false"]
        public List<string> StateCheckList = new();

        // ═══════════ 要触发的效果 ═══════════
        /// 当触发条件满足时，执行哪些效果
        /// 每个 TriggerEffect 包含效果索引 + 应用方式
        public List<TriggerEffect> TriggerEffects = new();
    }

    [Serializable]
    public class TriggerEffect
    {
        /// 效果在 Effects 列表中的索引
        public int EffectIndex;

        /// 效果应用方式
        public EffectApplyType EffectApplyType;
    }

    /// 效果应用方式
    public enum EffectApplyType
    {
        AllTargets,     // 对所有命中目标
        Creator,        // 对施法者自身
        FirstTarget,    // 仅对第一个目标
    }
}

TriggerTypeAutoTriggerType何时触发MOBA 实例
AutoTriggerInstantBuff 挂载时立即红 Buff 立即加攻速
AutoTriggerAction特定 ActionPoint 时荆棘甲：受到伤害后反伤
AutoTriggerCondition状态/时间条件满足时斩杀：目标 HP < 30% 时额外伤害

• Instant → 直接执行效果
• Action → 注册到 CombatEntity 的 ActionPoint 回调
• Condition → 创建定时器/条件监听器
• ManualTrigger → 由 ExecutionObject 的 Timeline Clip 触发

9.7 ExecutionObject — 执行时间轴配置

ExecutionObject 是 ScriptableObject，定义技能释放的时间轴（参见 Ch07 详细解析）。这里总结其配置相关字段。

ExecutionObject.cs — 配置字段摘要
[CreateAssetMenu(fileName = "Execution", menuName = "能力/Execution")]

    public float TotalTime;  // 执行体总时长（秒）

    // ═══════════ 时间轴片段 ═══════════

    [SerializeReference]
    public List<ExecutionClip> ExecutionClips = new();

    // ═══════════ Clip 类型 ═══════════
    // ExecutionClip 子类：
    // • SpawnItemClip      — 生成弹道/AOE 区域
    // • TriggerApplyClip   — 手动触发效果应用
    // • PlayAnimationClip  — 播放动画
    // • PlayEffectClip     — 播放特效
    // • PlayAudioClip      — 播放音效
}
ExecutionObject 与 AbilityConfigObject 的关系
AbilityConfigObject.ExecutionObject 引用了对应的 ExecutionObject。
• AbilityConfigObject 定义「这个技能有什么效果」
• ExecutionObject 定义「这些效果在什么时间以什么方式表现出来」

同一个 AbilityConfigObject 可以搭配不同的 ExecutionObject（如普通施法 vs 蓄力施法的不同时间轴）。

9.10 设计总结

Excel = 快速迭代
策划改数值不需要打开 Unity。Excel 编辑 → 一键导出 → 游戏内生效。数值平衡迭代效率极高。所有数值参数（CD、伤害系数、持续时间）都走这条路。
ScriptableObject = 可视化设计
效果组合、触发逻辑、时间轴编排 — 这些结构型数据在 Inspector 中可视化配置。技能设计师拖拖拽拽就能组合出复杂技能，不需要写代码。
Id 是唯一桥梁
两轨通过 Id 字段关联。Excel 中的 AbilityConfig.Id 必须与 ScriptableObject 中的 AbilityConfigObject.Id 相同。这是唯一的耦合点 — 简单但有效。
已知局限
• Id 关联靠人工保证一致性（无自动校验）
• Excel 导出是全量导出（改一行重新生成全部）
• ScriptableObject 无法热更新（需要 AssetBundle 方案）
• 无配置校验工具（填错公式只有运行时才报错）
