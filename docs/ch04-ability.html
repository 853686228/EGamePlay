Chapter 04

Ability — 技能/Buff 核心系统

EGamePlay 最核心的设计决策之一：技能和 Buff 是同一个东西。Ability 是统一的「能力」抽象，IsBuff 只是一个标志位。这消除了大量重复代码和特殊处理逻辑。

4.1 概述

核心定位
Ability 是 EGamePlay 战斗系统中最重要的 Entity 类型。它代表一个「能力」——可以是主动技能、被动技能、Buff、Debuff、光环等任何可以挂载到战斗实体上的行为声明。

关键洞察：技能和 Buff 在数据结构上完全一致，差异仅在于激活方式和持续时间管理。统一它们是这套框架最精妙的设计决策。

4.2 Ability 实体结构

Ability 继承自 EcsEntity，是一个轻量级的数据容器。它持有配置引用、运行时状态、以及子实体列表（Effects 和 Triggers）。

Ability.cs — 完整源码
namespace EGamePlay.Combat
{
    public partial class Ability : EcsEntity
    {
        // ═══════════ 所有权 ═══════════
        public CombatEntity OwnerEntity { get { return GetParent<CombatEntity>(); } set { } }
        public EcsEntity ParentEntity { get => Parent; }

        // ═══════════ 配置数据 ═══════════
        public AbilityConfig Config { get; set; }              // 运行时从表加载的配置
        public AbilityConfigObject ConfigObject { get; set; }  // ScriptableObject 配置

        // ═══════════ 运行时状态 ═══════════
        public bool Spelling { get; set; }                     // 是否正在施法
        public GameTimer CooldownTimer { get; } = new GameTimer(1f);  // 冷却计时器
        public ExecutionObject ExecutionObject { get; set; }  // 执行时间轴配置

        // ═══════════ 类型判断 ═══════════
        public bool IsBuff => Config.Type == "Buff";            // ★ 只是一个属性判断
        public bool IsSkill => !IsBuff;

        // ═══════════ 名称 ═══════════
        public string Name { get; set; }

        // ═══════════ 子实体集合 ═══════════
        public List<AbilityEffect> AbilityEffects { get; private set; }
            = new List<AbilityEffect>();
        public List<AbilityTrigger> AbilityTriggers { get; private set; }
            = new List<AbilityTrigger>();
    }
}
设计洞察 — 数据结构极简
Ability 本身只有 30 行代码。所有属性都是直球声明，没有任何逻辑方法。这正是 ECS「数据与逻辑分离」的体现 — 所有行为在 AbilitySystem 中。

OwnerEntity 通过 GetParent<CombatEntity>() 获得，利用了 EcsNode 的层级关系，而不是额外存储一个引用。

属性用途一览

属性类型来源说明
ConfigAbilityConfigAwake 时从表加载运行时配置（Type、TargetGroup 等）
ConfigObjectAbilityConfigObjectAttach 时注入ScriptableObject，含 Effects/Triggers 列表
Spellingbool运行时设置施法过程中为 true，防止重复施放
CooldownTimerGameTimer创建时初始化冷却倒计时

4.3 AbilitySystem.Awake — 初始化流程

Awake 是 Ability 创建后立刻执行的初始化逻辑。它从配置表加载数据、解析目标规则、创建子实体（Effect/Trigger），最后加载执行时间轴资源。

AbilitySystem.cs — Awake
public void Awake(Ability entity)
{
    // ① 从配置表加载 AbilityConfig
    entity.Config = AppStatic.Config.Get<AbilityConfig>(entity.ConfigObject.Id);
    entity.Name = entity.Config.Name;

    // ② 解析目标阵营类型
    CheckAffectTargetType(entity);

    // ③ 解析目标选取方式
    CheckTargetSelectType(entity);

    // ④ 创建所有 AbilityEffect 子实体
    foreach (var item in entity.ConfigObject.Effects)
    {
        var abilityEffect = AbilityEffectSystem.Create(entity, item);
        abilityEffect.Init();
        entity.AbilityEffects.Add(abilityEffect);
    }

    // ⑤ 创建所有 AbilityTrigger 子实体
    foreach (var item in entity.ConfigObject.TriggerActions)
    {
        var abilityTrigger = AbilityTriggerSystem.Create(entity, item);
        abilityTrigger.Init();
        entity.AbilityTriggers.Add(abilityTrigger);
    }

    // ⑥ 加载执行时间轴
    LoadExecution(entity);
}

CheckAffectTargetType — 目标阵营解析

AbilitySystem.cs — 阵营映射
private static void CheckAffectTargetType(Ability entity)
{
    if (entity.Config.TargetGroup == "己方")
    {
        entity.ConfigObject.AffectTargetType = SkillAffectTargetType.SelfTeam;
    }
    else if (entity.Config.TargetGroup == "敌方")
    {
        entity.ConfigObject.AffectTargetType = SkillAffectTargetType.EnemyTeam;
    }
    else if (entity.Config.TargetGroup == "自身")
    {
        entity.ConfigObject.AffectTargetType = SkillAffectTargetType.Self;
    }
    else
    {
        Log.Error($"技能目标阵营错误 {entity.Config.TargetGroup}");
    }
}

CheckTargetSelectType — 目标选取方式

AbilitySystem.cs — 选取映射
private static void CheckTargetSelectType(Ability entity)
{
    if (entity.IsSkill)  // ← 只有技能需要设置选取方式，Buff 直接作用于持有者
    {
        if (entity.Config.TargetSelect == "碰撞检测")
        {
            entity.ConfigObject.TargetSelectType = SkillTargetSelectType.CollisionSelect;
        }
        else if (entity.Config.TargetSelect == "条件指定")
        {
            entity.ConfigObject.TargetSelectType = SkillTargetSelectType.ConditionSelect;
        }
        else if (entity.Config.TargetSelect == "手动指定")
        {
            entity.ConfigObject.TargetSelectType = SkillTargetSelectType.PlayerSelect;
        }
        else
        {
            Log.Error($"目标选取类型错误 {entity.Config.TargetSelect}");
        }
    }
}
中文字符串作为配置键
注意 "己方"、"碰撞检测" 等使用中文字符串匹配。这来自策划的 Excel 配置表。好处是策划可以直接阅读和编辑；坏处是拼写错误无编译期保护。框架通过 Log.Error 在运行时捕获不匹配的情况。

LoadExecution — 加载时间轴

AbilitySystem.cs — 资源加载
public static void LoadExecution(Ability entity)
{
    entity.ExecutionObject = AssetUtils.LoadObject<ExecutionObject>(
        $"{AbilityManagerObject.ExecutionResFolder}/Execution_{entity.ConfigObject.Id}");
    if (entity.ExecutionObject == null)
    {
        return;  // Buff 通常没有执行时间轴，这是正常的
    }
}
Awake 初始化的 6 步设计
Awake 的步骤顺序是精心设计的：
① Config 必须最先加载，因为后续步骤都依赖它
② ③ 解析配置中的字符串为枚举（将不确定性在初始化时解决）
④ ⑤ 创建子实体（Effect 和 Trigger 需要读取 Config 数据）
⑥ LoadExecution 最后执行，因为它可能失败（Buff 没有 Execution），且不影响前面的步骤

4.4 生命周期管理

AbilitySystem 实现了 IEnable、IDisable、IUpdate 三个生命周期接口，控制 Ability 的激活/暂停/持续运行。

Enable — 激活

AbilitySystem.cs — Enable
public void Enable(Ability entity)
{
    // 级联启用所有 Effect
    foreach (var item in entity.AbilityEffects)
    {
        item.Enable = true;
    }
    // 级联启用所有 Trigger
    foreach (var item in entity.AbilityTriggers)
    {
        item.Enable = true;
    }
}

Disable — 禁用

AbilitySystem.cs — Disable
public void Disable(Ability entity)
{
    // 级联禁用所有 Effect
    foreach (var item in entity.AbilityEffects)
    {
        item.Enable = false;
    }
    // 级联禁用所有 Trigger
    foreach (var item in entity.AbilityTriggers)
    {
        item.Enable = false;
    }
}
级联的双重含义
当 Ability.Enable = true 时，EcsNode 首先触发 AbilitySystem.Enable()（上面的代码），它显式启用 Effects 和 Triggers。
而 EcsEntity 的 Enable setter 本身也会级联启用所有 Components（参见 Ch01）。

效果：一行 ability.Enable = true 同时启用了 Effects + Triggers + Components 三个维度。

Update — 持续时间追踪

AbilitySystem.cs — Update
public void Update(Ability entity)
{
    if (entity.GetComponent<AbilityDurationComponent>() is { } component)
    {
        AbilityDurationSystem.Update(entity, component);
    }
}

AbilityDurationSystem — 持续时间管理

AbilityDurationSystem.cs — 完整源码
public class AbilityDurationSystem : AComponentSystem<Ability, AbilityDurationComponent>,
    IAwake<Ability, AbilityDurationComponent>
{
    public void Awake(Ability entity, AbilityDurationComponent component)
    {
        var lifeTime = component.Duration;
        component.LifeTimer = new GameTimer(lifeTime);
    }

    public static void Update(Ability entity, AbilityDurationComponent component)
    {
        if (component.LifeTimer.IsRunning)
        {
            component.LifeTimer.UpdateAsFinish(Time.deltaTime, () => OnTimeEnd(entity));
        }
    }

    public static void OnTimeEnd(Ability ability)
    {
        var config = ability.Config;
        if (config.Type == "Skill")
        {
            SkillSystem.Remove(ability.ParentEntity as CombatEntity, ability);
        }
        if (config.Type == "Buff")
        {
            BuffSystem.Remove(ability.ParentEntity as CombatEntity, ability);
        }
    }
}
持续时间是可选的
AbilityDurationComponent 只在需要时挂载。被动技能永远不挂它（永久生效），Buff 在 Attach 时根据配置挂载。
AbilitySystem.Update 用 is { } component 模式匹配安全获取 — 如果没有这个组件，整个 Update 就是空操作。
零特殊情况，纯组合式设计。

4.5 Attach / Remove — 挂载与移除

Ability 通过 AbilitySystem.Attach 挂载到 CombatEntity 上，通过 AbilitySystem.Remove 移除。这是外部代码与 Ability 系统交互的唯一入口。

AbilitySystem.cs — Attach
/// 挂载能力，技能、被动、buff等都通过这个接口挂载
public static Ability Attach(CombatEntity entity, object configObject)
{
    // ① 获取 AbilityComponent（存放所有 Ability 的容器）
    var component = entity.GetComponent<AbilityComponent>();

    // ② 创建 Ability 子实体，在 Awake 前注入 ConfigObject
    var ability = entity.AddChild<Ability>(x => x.ConfigObject = configObject as AbilityConfigObject);
    // ↑ AddChild 触发 AbilitySystem.Awake → 加载配置、创建子实体

    // ③ 添加等级组件
    ability.AddComponent<AbilityLevelComponent>();

    // ④ 手动触发 Init（延迟初始化）
    ability.Init();

    // ⑤ 注册到 AbilityComponent 的 ID 映射表
    component.IdAbilities.Add(ability.Id, ability);

    return ability;
}

AbilitySystem.cs — Remove
public static void Remove(CombatEntity entity, Ability ability)
{
    // ① 从注册表移除
    var component = entity.GetComponent<AbilityComponent>();
    component.IdAbilities.Remove(ability.Id);

    // ② 结束能力（Disable + Destroy）
    EndAbility(ability);
}

public static void EndAbility(Ability entity)
{
    DeactivateAbility(entity);   // Enable = false → 触发所有 IDisable
    EcsObject.Destroy(entity);    // 销毁 Entity + 递归销毁子树
}
Attach 执行序列详解
Attach(entity, config) 一调用，以下事件按顺序发生：

1. AddChild<Ability>() → 创建 Ability 实例，设置 Parent
2. initAction 执行 → ConfigObject = configObject
3. EcsNode 触发 AbilitySystem.Awake() → 加载 Config、创建 Effects/Triggers
4. AddComponent<AbilityLevelComponent>() → 挂载等级组件
5. ability.Init() → 触发 IInit System（如有）
6. 注册到 IdAbilities 字典

注意 Awake 在第 3 步，此时 ConfigObject 已在第 2 步设好。initAction 的存在确保了 Awake 能安全读取配置。

ActivateAbility / DeactivateAbility

AbilitySystem.cs — 激活/停用
public static void TryActivateAbility(Ability entity)
{
    ActivateAbility(entity);
}

public static void ActivateAbility(Ability entity)
{
    entity.Enable = true;   // 触发 AbilitySystem.Enable → 级联启用
}

public static void DeactivateAbility(Ability entity)
{
    entity.Enable = false;  // 触发 AbilitySystem.Disable → 级联禁用
}

4.6 Skill vs Buff — 统一设计

这是整个框架最重要的架构决策：技能和 Buff 使用完全相同的 Ability 类。差异仅通过配置和组件组合来区分。

技能 (Skill)
• Config.Type == "Skill"
• 有 ExecutionObject（技能表现时间轴）
• 有目标选取（CollisionSelect / PlayerSelect）
• 通常通过 Trigger 触发 Effect
• 可能有冷却（CooldownTimer）
• 可能有持续时间（引导型技能）
Buff / Debuff
• Config.Type == "Buff"
• 通常无 ExecutionObject
• 目标 = 自身（TargetGroup = "自身"）
• Enable 时 Effect 直接生效（属性修改/行为控制）
• 几乎总有 AbilityDurationComponent
• 持续时间结束自动 Remove 
统一机制 — IsBuff 只是一个标志
// Ability.cs 中只有这一行区分 Skill 和 Buff：
public bool IsBuff => Config.Type == "Buff";

// AbilityDurationSystem.OnTimeEnd — 结束时根据类型调用不同的 Remove
if (config.Type == "Skill")
    SkillSystem.Remove(ability.ParentEntity as CombatEntity, ability);
if (config.Type == "Buff")
    BuffSystem.Remove(ability.ParentEntity as CombatEntity, ability);

// SkillSystem.Remove 和 BuffSystem.Remove 最终都调用 AbilitySystem.Remove
// 区别仅在于它们各自的事件通知（如 UI 刷新、Buff 图标移除等）
为什么这个设计是对的
传统做法是 Skill 和 Buff 各有一套类体系，导致：
• 伤害效果写两套（技能伤害 + Buff 伤害）
• 属性修改写两套
• 持续时间管理写两套
• 触发条件写两套

统一为 Ability 后，所有这些都只需要一套实现。Buff 给目标加状态？那就是一个 EffectAddBuff，和技能施加 Buff 走完全相同的代码路径。

Linus 会说：「好的设计消除特殊情况，而不是增加 if 分支。」

实际代码路径对比

行为技能的路径Buff 的路径

激活手动调用 ActivateAbility挂载后立即 Enable

4.8 设计总结

统一抽象的代价
Skill 和 Buff 共享一个类意味着 Ability 上可能存在对当前类型无意义的属性（如 Buff 的 CooldownTimer 永远不用）。这点内存浪费换来的是代码复用和架构简洁，是值得的。
组合式能力声明
一个 Ability 的行为完全由它的 Effects 和 Triggers 列表决定。同样的 EffectDamageComponent 可以出现在技能的 Effect 上，也可以出现在 Buff 的 Effect 上。零代码重复。
Enable 是核心机制
整个系统的激活/停用都通过 Enable 属性驱动。一行 ability.Enable = false 触发完整的回退链：Disable System → 移除 Modifier → 数值恢复。极简的 API，强大的效果。
已知局限
• 字符串类型判断（"Skill"/"Buff"）缺少编译期安全
• 没有 Ability 堆叠机制（同名 Buff 叠加需要额外处理）
• AbilityDurationSystem 通过手动 Update 调用而非 EcsNode 自动驱动
